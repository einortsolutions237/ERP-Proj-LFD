# LFD Services — Enterprise ERP

## What
Enterprise ERP for LFD Services (starting as a single location, architected for multi-branch growth). Supersedes the earlier "MediCore ERP" healthcare-domain scope — this is a business/retail-and-services enterprise design spanning POS, inventory, CRM, accounting, and HR.

Stack: Next.js (App Router) + TypeScript + Tailwind CSS, Firebase Auth + Firestore + Cloud Functions, deployed on Vercel. Firebase project: `erp-lfd` (migrated from `lfd-erp-4713b` on 2026-07-03 — the old project had an unspecified account-level problem; all Firestore data and the one real Auth account were copied over preserving doc IDs/UID, and the old project was left untouched as a fallback rather than deleted).

Architecture: modular monolith, not microservices — clean module boundaries inside one Next.js app. Every record carries a `branchId` and every permission check is branch-scoped, starting from Phase 1, even though only one branch exists today.

## Why
LFD Services needs one system to replace fragmented/manual operations — starting with staffing and access control, extending to POS, inventory, CRM, accounting, and HR. Full module vision: `docs/project-brief.md`.

## How

**Permissions** — 6 roles: super_admin (protected, immutable, cannot be demoted or deleted in-app), admin, branch_manager, hr_admin, finance_admin, it_admin. Permissions are mapped per module (POS, inventory, CRM, accounting, HR) — not by role alone — and enforced at three layers: UI, server, Firestore rules, never just one. Firebase custom claims (`role`, `branchId`) are the sole source of truth for authorization; Firestore is profile metadata only and is never read to make an access-control decision.

**Never repeat these — each caused a real production issue in an earlier build of this project:**
- No Proxy-based Firebase SDK initialization — use direct SDK calls
- No hardcoded Firebase credentials or fallback values in client code — env vars only, no `??` fallback to literals
- Don't let tooling silently regenerate/empty `tsconfig.json` — verify it after any AI-assisted scaffold
- No `Math.random()` for temporary passwords or any security-sensitive value — use a CSPRNG
- Firestore rules default-deny; every collection needs an explicit, tested rule — no blanket permissive rules
- Session cookies: don't hardcode `secure: true` + `sameSite: none` — it silently drops cookies in non-HTTPS preview environments
- No public self-registration route — accounts are created by a permitted internal role only
- Cloud Functions v2, not v1 (deprecated)

**Audit log** — every login, failed login, logout, and create/edit/delete on customers, staff, or permissions gets an immutable log entry, written server-side only. Never expose an edit or delete path for audit entries.

**Login has two paths, chosen server-side by account role, not by the client:**
- `super_admin` and `admin` (the roles where a failed-login attempt is the highest-value security signal) verify the password server-side against the Identity Toolkit REST API. Every attempt — success or failure — is written to the audit log by code we control; this path is tamper-proof.
- `branch_manager`, `hr_admin`, `finance_admin`, `it_admin` sign in via the Firebase client SDK directly. A failed attempt never reaches our server on its own — the client reports it best-effort after the fact. Treat these `login_failed` entries as telemetry, not a tamper-proof control; this is an accepted trade-off, not a gap to silently fix later.
- The routing decision (which path a given email uses) is made by a pre-flight server lookup of the account's role before any password is checked — this necessarily reveals whether a given email belongs to a `super_admin`/`admin` account (a narrow account-role signal, not full account-existence enumeration — unknown emails and non-strict-role emails get an identical response). Accepted for an internal-only ERP; revisit if this app ever gets external-facing accounts.

**Catalog & stock ownership** — `products` and `services` are org-wide catalog entries, not branch-scoped (a SKU/price is the same everywhere). What's branch-specific is *quantity on hand*: `productStock` holds one doc per branch+product, and its `quantity` is a derived, server-maintained total — never directly settable. Every stock change writes an immutable `stockMovements` ledger entry (restock/adjustment/waste/transfer_out/transfer_in/sale/void), and the same Admin-SDK write path that creates the movement atomically increments `productStock.quantity` in the same transaction. No edit form ever sets quantity directly.

**POS & sales** — `sales` is branch-scoped (a sale physically happens at one location). Line-item `name`/`unitPrice` are snapshotted server-side at sale time from the current catalog — never trusted from the client and never a live reference back to `products`/`services` (a later price change must not retroactively alter a completed sale). Payment (`cash`/`mtn_momo`/`orange_money`) is a manually-entered record only — the system never calls out to MTN, Orange, or any payment processor to verify it. Selling a product decrements `productStock` and writes a `stockMovements` entry (`type: 'sale'`) via the same transaction discipline as every other stock change; selling a service touches no stock. `cashier` is a new, narrow role (client-SDK login path, `pos.sale.create`/`pos.sale.view` only, zero admin/inventory/supplier access anywhere) — from Phase 5, `cashier` also has `hr.leave.request`/`hr.attendance.self` like every other role, since those aren't module-specific access, they're an employee's own self-service actions.

**Voiding a sale** — full-sale void only; no partial/line-item returns, no payment reversal (the system never verified payment methods in the first place, so there's nothing on that side to undo). Voiding is additive, not destructive: the original `sales` doc's `lineItems`/`payments`/`total` are never touched — only `voidedAt`/`voidedBy`/`voidReason` are set, once, on top, and a second void attempt on an already-voided sale is rejected. Stock reversal is ledger-driven, not re-derived from `lineItems`: for each `type:'sale'` `stockMovements` entry the original sale created, a new `type:'void'` entry (positive delta, same `saleId` link) is written in the same transaction that sets the sale's voided fields — the ledger is the source of truth for what to reverse, not a re-aggregation of the sale's line items. `pos.sale.void` is `branch_manager`/`admin`/`super_admin` only, never `cashier`; a `branch_manager` may only void a sale in their own branch.

**Org-wide entities, and how that principle generalizes** — `products`, `services`, `suppliers`, and (from Phase 4) `customers` are all org-wide catalog/reference entries, not branch-scoped: the same customer, product, or supplier record is the same one everywhere, never duplicated per branch. Each may carry a `*BranchId`-style field (e.g. `registeredBranchId` on a customer, matching the role `supplierId`/similar reference fields play elsewhere) for *reference only* — which branch first touched the record — never for access control or filtering; every role reads the full org-wide set. What's branch-specific is downstream activity tied to a branch (`productStock` quantities, `sales` themselves) — the entity and the branch-scoped activity referencing it are two different things, and only the latter is ever filtered by `branchId`.

**HR — leave & attendance** — `leaveRequests` and `attendanceRecords` are branch-scoped, not org-wide (a leave request or a clock-in belongs to wherever the staff member currently works; `branchId` is copied from the requester's custom claims at request/check-in time — the same claims-are-truth source every other branch check in this app already uses). `hr.leave.request` and `hr.attendance.self` are the first capabilities granted to every role rather than a role-group subset — any staff member requests their own leave and clocks themselves in/out, `cashier` included. Self-approval is blocked by an explicit server-side check (`existing.staffId === user.uid`), not a UI omission — enforced regardless of role, including a `super_admin` reviewing their own request. Approver visibility is a deliberate, new exception to this app's usual branch-scoping: `branch_manager` sees only their own branch's requests/attendance (the same restriction already enforced on `pos.sale.void`), but `hr_admin`/`admin`/`super_admin` see every branch — the first place in the app where a non-`branch_manager` role's reads span branches, because HR review is inherently an org-wide function, unlike a branch-scoped sale. Both collections' Firestore rules are fully closed (`allow read, write: if false`, matching `rateLimits`) rather than getting the open-read treatment `products`/`customers`/`sales` get — the access rule here (own record, OR approver with branch-scoping but never on your own record) is exactly the kind of role logic this project keeps out of `firestore.rules` and enforces server-side instead. "Today" for attendance is the server's **UTC calendar date** — a deliberately documented temporary stand-in (single source of truth: `getTodayDateString()`) until per-branch timezones exist as a real concept in this app; revisit alongside the jurisdiction decision. Approving a leave request also checks it against the requester's other already-`approved` requests for date overlap, inside the same transaction that writes the approval — the same "read-check-write atomic" discipline as the phone-uniqueness and sale-void transactions, just comparing date ranges in application code rather than a Firestore range query.

**Notifications are Cloud-Function-driven, not inline in the routes that cause them** — `api/sales/route.ts`, `api/stock/movements/route.ts`, `api/stock/transfer/route.ts`, and `api/leave-requests/*` have all been through clean, sometimes Opus-tier, audits; adding a notification-creation call inside any of them would mean re-touching an already-verified transaction's diff surface for an unrelated concern. Instead, a Firestore-triggered Cloud Function (v2 API only, per the existing "Never repeat these" rule) reacts to what those routes already write — `stockMovements` creates, `leaveRequests` creates/updates — and creates `notifications` docs from the side. This generalizes past Phase 6: any future side effect of an already-audited write (a report row, a downstream sync, another notification type) is a new trigger reacting to the existing write, not a new line inside the write itself. `notifications` are personal, not role/branch-scoped data — access is ownership-only (`recipientUid == request.auth.uid`), so unlike every other module, there's no `Capability` entry for them at all; a bare authenticated check is the only gate, enforced at the API layer, with Firestore rules fully closed the same way as `leaveRequests`/`attendanceRecords`. Cloud Functions run against this project's explicitly-named `"default"` Firestore database, not the SDK's implicit `"(default)"` — see `src/lib/firebase/admin.ts`'s comment on this; a v2 Firestore trigger must pass `database: 'default'` in its options or it silently never fires. Notification docs get a deterministic ID (`${type}_${eventId}_${recipientUid}`) and are written with `create()`, never `set()` — Cloud Functions triggers are at-least-once, not exactly-once, so a retry of the same event must fail loudly on the already-existing doc (swallowed as a known-harmless case) rather than silently re-write it.

**Reporting reads on demand — no new collections, no materialized aggregates** — Phase 7's sales/revenue and stock-level reports compute everything at request time from `sales`, `stockMovements`/`productStock`, `products`, and `services`, the same already-audited data every other module already writes. This is the read-only counterpart to Notifications' "react to the write, don't touch the write" principle: reporting doesn't even need a Cloud Function, since there's no write to react to — it's a new query, not a new side effect. A report API route's output is always plain aggregated primitives (numbers, strings, already-summed totals) — never a raw Firestore doc or Timestamp passed through — which sidesteps the Timestamp-leak bug class this project has hit twice by construction, not by discipline. Line-item names/prices come straight off `sales.lineItems`'s own snapshot (never re-joined against current `products`/`services`), matching the existing "a later price change must not retroactively alter a completed sale" rule — a report is not exempt from that just because it's read-only. The low-stock threshold comparison (`quantity <= reorderThreshold`) is defined once (`src/lib/inventory/lowStock.ts`) and reused by both the report and the existing Cloud Function trigger (`functions/src/lowStock.ts`) — but since `functions/` is a separate deployable that can't import from the root `src/` tree (the same boundary that already forced `APPROVER_ROLES` to be duplicated there), the comparison itself is duplicated with a "keep in sync" comment, not literally shared code; only the app-side report and any other app-side reads share the real helper.
