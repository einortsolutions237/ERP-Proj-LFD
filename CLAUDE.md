# LFD Services — Enterprise ERP

## What
Enterprise ERP for LFD Services (starting as a single location, architected for multi-branch growth). A business/retail-and-services enterprise design spanning POS, inventory, CRM, accounting, and HR — LFD Services sells both physical products and services, paid by cash, MTN Mobile Money, or Orange Money.

Stack: Next.js (App Router) + TypeScript + Tailwind CSS, Firebase Auth + Firestore + Cloud Functions, deployed on Vercel. Firebase project: `erp-lfd` (confirmed against `.env.local`/`.firebaserc` — `lfd-erp-4713b` is the pre-migration project, retired 2026-07-03; this is the second time a pasted CLAUDE.md copy has reverted to the old project ID, see the CLAUDE.md-staleness note in memory).

Architecture: modular monolith, not microservices — clean module boundaries inside one Next.js app. Every record carries a `branchId` and every permission check is branch-scoped, starting from Phase 1. This was designed against a single branch through Phase 7 and first exercised with a second one in Phase 8, which found the same bug in three places, not two: staff and department *creation*, and — caught mid-phase, easy to miss in a quick audit — department *edit/delete* (`PATCH`/`DELETE /api/departments/[id]`) had it too. All three fixed the same way, via a shared `isBranchLocked(role)` helper (`branch_manager`/`cashier` are branch-locked; everyone else isn't): a branch-locked role's own `branchId` is used unconditionally, an explicit `branchId` in the request body is only honored for non-branch-locked roles, and it's validated against a real `branches` doc before being accepted. Suppliers, customers, products, services, and branches were all checked and correctly confirmed org-wide, needing no such fix — every route that touches `branchId` on read or write now has a documented conclusion, not just the ones that needed changing. Any new route creating a branch-scoped record should use `isBranchLocked` rather than re-deriving the role list.

**Resolved in Phase 20 (was an open product question)** — `GET /api/sales` (and the `/pos/sales` page) previously filtered by `user.branchId` unconditionally for every role. The decision was made to match the rest of the app: both now scope by `isBranchLocked(user.role)` (Phase 12's pattern) — branch-locked roles (`branch_manager`/`cashier`) still see only their own branch's sales, while org-wide roles see every branch's, the same as staff/departments/stock. Note that after Phase 17 only `super_admin`/`branch_manager`/`cashier` hold `pos.sale.view` at all (`admin`/`general_manager`/`finance_admin` 403 regardless), so in practice `super_admin` is the role the change actually broadens. `GET /api/reports/sales` was deliberately **not** touched — it still scopes only `branch_manager` to its own branch (via `role === 'branch_manager'`, not `isBranchLocked`), unchanged. Live-verified in Phase 20 (see its completion report).

**Fixed and audited (Phase 12)**: `staff/page.tsx`, `departments/page.tsx`, `stock/page.tsx`, and a fourth instance the systematic sweep found, `roles/page.tsx`, all now use `isBranchLocked()` matching their sibling routes exactly, instead of the hardcoded branch filter that used to bypass Phase 8's fix entirely. Fixing stock's read also surfaced and fixed a related latent bug: `StockTable.tsx` was passing the *viewer's own* ambient `branchId` into every row's Adjust/Transfer form regardless of that row's actual branch — harmless while the read bug hid multi-branch rows, live-reachable the moment it didn't. Each row now carries and uses its own `branchId` throughout, including in the transfer destination-exclusion logic, which was a genuine correctness improvement, not just a rewrite (an org-wide admin viewing a foreign branch's row can now correctly transfer stock into their own branch, previously wrongly excluded). Minor, pre-existing, unrelated finding surfaced along the way: `stock/page.tsx`'s low-stock indicator uses `quantity < reorderThreshold` while the shared `isLowStock()` (used by reports and the Cloud Function) uses `<=` — boundary-only, fix whenever something else touches that file by switching to the shared helper instead of its own inline check. `pos/sales/page.tsx` mirrors `GET /api/sales`'s own unconditional branch filter and was correctly left alone — that route was never given the Phase 8 fix in the first place, it's the still-open product question below, not a bug with a known pattern to restore.

**LFD Services is a hybrid business, confirmed** — the retail/services operation already built, plus a genuine clinical operation, same people: a retail customer can also become a patient. This is new as of the clinical-module conversation, not something the original 12 phases were built against. Core design decision: patients are not a new entity — they're the existing `customers` collection, with clinical data (treatment/visit records, lab results, seminar attendance) living in separate, strictly access-walled collections referenced by `customerId`, not fields on the customer doc itself. A person becomes a "patient" the moment a clinical record exists for them; nothing else about their existing commercial record changes. Clinical data is visible to `doctor`, `medical_secretary` (Phase 13.1, full read like doctor, no authoring), `general_manager` (full read like doctor/medical_secretary, no authoring — explicit override, general_manager was originally proposed as excluded, corrected by direct instruction), and `super_admin` only — not `admin` despite being broad elsewhere, not `branch_manager`, `cashier`, `hr_admin`, `finance_admin`, or `it_admin`. Every read is still audit-logged regardless of who's reading — broadening who holds the capability doesn't broaden it without a trail. `medical_secretary` also gets `crm.customer.view` (payment/purchase history) — the first role that legitimately spans both the commercial and clinical sides of the wall, since that's what the job actually is, not a breach of it. Treatment/consultation payment reuses the existing POS flow entirely — a consultation is a `service` in the existing catalog, sold through the already-audited checkout — so payment history needs no new infrastructure, only the clinical record itself, appointments, lab, and seminars are genuinely new. Appointments are new and specific to clinical visits — they coexist with, not replace, the confirmed walk-in model for retail/spa-style services; a doctor visit is the one appointment-based exception, not a reversal of the Phase 2 decision. Roadmap: Phase 13 (clinical foundation — doctor role, the patient data wall, treatment/visit records; complete, see below), Phase 13.1 (`medical_secretary` role; complete, see below), Phase 14 (appointments, booked against a specific doctor; complete, see below), Phase 15 (laboratory — test orders and results; complete, see below), Phase 16 (health seminars — events, `protocol` records attendance, `medical_secretary` gets the oversight view; complete, see below). The clinical module as originally scoped (Phases 13-16) is complete.

## Why
LFD Services needs one system to replace fragmented/manual operations — starting with staffing and access control, extending to POS, inventory, CRM, accounting, and HR. Full module vision: `docs/project-brief.md`.

## How

**Permissions** — 7 roles: super_admin (protected, immutable, cannot be demoted or deleted in-app), admin, branch_manager, hr_admin, finance_admin, it_admin, cashier (added Phase 3 — frontline POS role, no admin/inventory/supplier access anywhere). Permissions are mapped per module (admin, POS, inventory, CRM, accounting, HR) — not by role alone — and enforced at three layers: UI, server, Firestore rules, never just one. Firebase custom claims (`role`, `branchId`) are the sole source of truth for authorization; Firestore is profile metadata only and is never read to make an access-control decision. `branch_manager` has full catalog-adjacent access (suppliers, stock) but not `inventory.catalog.manage` itself — product/service definitions are admin/super_admin only, a deliberate split from Phase 2, not an oversight.

**Never repeat these — each caused a real production issue in an earlier build of this project:**
- No Proxy-based Firebase SDK initialization — use direct SDK calls
- No hardcoded Firebase credentials or fallback values in client code — env vars only, no `??` fallback to literals
- Don't let tooling silently regenerate/empty `tsconfig.json` — verify it after any AI-assisted scaffold
- No `Math.random()` for temporary passwords or any security-sensitive value — use a CSPRNG
- Firestore rules default-deny; every collection needs an explicit, tested rule — no blanket permissive rules
- Session cookies: don't hardcode `secure: true` + `sameSite: none` — it silently drops cookies in non-HTTPS preview environments
- No public self-registration route — accounts are created by a permitted internal role only
- Cloud Functions v2, not v1 (deprecated)
- No unauthenticated, unthrottled endpoint that checks a password or secret — rate-limit it (Phase 1 audit finding, fixed in Phase 2: see `src/lib/auth/rate-limit.ts`, Firestore-backed, transaction-safe, per-email and per-IP)
- Any read-check-write sequence on a numeric quantity needs a real Firestore transaction, not a plain read followed by a separate write (stock movements/transfers do this correctly; this applies to POS stock decrements too — reuse that exact pattern, don't write a parallel one)

**Known-issues policy** — check `docs/tech-debt.md` and this file before starting a phase. If the phase's own work already touches the same file or area as something already flagged, fix it as part of that work and say so in the plan and the completion report. If it doesn't touch that area, just re-flag it rather than going out of scope to chase it down — a phase stays bounded to what it was asked to build, it doesn't turn into a standing bug hunt. **TD-1 resolved in Phase 20**: `product_edit`/`service_edit`/`supplier_edit` now write `details: { before, after }` capturing only the changed fields (each route already read the existing doc, so no new read was added), and `customer_create` writes its full creation payload — both live-verified. **TD-2 resolved in Phase 20**: every `stockMovements` write from a sale/adjustment/transfer now snapshots `resultingQuantity` inside the same transaction, and `onLowStock` reads it directly instead of reconstructing "quantity before this movement" from a live `productStock` read — the race window is closed (proven live by forcing two rapid same-product movements; the crossing one reported its own `resultingQuantity`, not the polluted live value). The `void` reversal path was deliberately left untouched (its always-positive delta can't trip the crossing check), so its movement docs carry no `resultingQuantity` and `onLowStock` falls back to the original reconstruction only for those and pre-phase historical docs. **TD-3 resolved in Phase 15**: `DELETE /api/customers/[id]` now blocks independently on `sales`, `treatments`, `appointments`, and `labOrders`, each verified live — no longer open. The standing question of whether this allowlist-of-dependent-collections shape should eventually become a soft-delete/archive model once a fifth dependent collection exists (Phase 16's seminar attendance is the next candidate) remains genuinely open, not resolved — worth deciding before that phase, not after.

**Notifications are Cloud-Function-driven, not bolted into existing routes** — confirmed working as intended in Phase 6: the four already-audited routes it could have touched instead came out unmodified, verified directly. A Firestore trigger listens for writes that already happen (`stockMovements`, `leaveRequests`) and creates a `notifications` doc reactively. Every notification write uses `create()` against a deterministic doc ID, not `set()` — Cloud Functions redeliver events at-least-once, and `create()` makes a retry fail loudly (caught and swallowed) instead of silently double-writing.

**Reporting reads on-demand, no new collections** — a report is computed at request time from `sales`/`stockMovements`/`productStock`, not maintained as a running aggregate the way `productStock.quantity` is. Fine at current data volume; revisit toward pre-aggregated summary docs only if a report route becomes slow as history grows — don't build that machinery preemptively. Revenue figures always exclude voided sales (`voidedAt != null`) from the totals, but a report should still surface how many sales were voided and their value alongside the real total, not hide them entirely.

**Audit log** — every login, failed login, logout, and create/edit/delete on customers, staff, catalog/stock, sales, or permissions gets an immutable log entry, written server-side only. Never expose an edit or delete path for audit entries. Known gap (tracked in `docs/tech-debt.md` as TD-1): catalog edit actions (`product_edit`/`service_edit`/`supplier_edit`) record that a change happened but not what changed. Stock, sales, and permission-change actions carry full detail in `details` — sale creation should follow that same fuller pattern (line items, total, payment breakdown), not the bare catalog-edit one.

**Login has two paths, chosen server-side by account role, not by the client:**
- `super_admin` and `admin` (the roles where a failed-login attempt is the highest-value security signal) verify the password server-side against the Identity Toolkit REST API. Every attempt — success or failure — is written to the audit log by code we control; this path is tamper-proof.
- `branch_manager`, `hr_admin`, `finance_admin`, `it_admin`, and `cashier` sign in via the Firebase client SDK directly. A failed attempt never reaches our server on its own — the client reports it best-effort after the fact. Treat these `login_failed` entries as telemetry, not a tamper-proof control; this is an accepted trade-off, not a gap to silently fix later. `cashier` belongs on this path deliberately — high-frequency, low-privilege logins are exactly what it's for.
- The routing decision (which path a given email uses) is made by a pre-flight server lookup of the account's role before any password is checked — this necessarily reveals whether a given email belongs to a `super_admin`/`admin` account (a narrow account-role signal, not full account-existence enumeration — unknown emails and non-strict-role emails get an identical response). Accepted for an internal-only ERP; revisit if this app ever gets external-facing accounts. This signal is also visible via response timing, not just response shape, since the strict path makes an extra network call before responding — same accepted trade-off, second channel.

**Catalog & stock ownership** — `products` and `services` are org-wide catalog entries, not branch-scoped (a SKU/price is the same everywhere). What's branch-specific is quantity on hand: `productStock` holds one doc per branch+product, and its `quantity` is a derived, server-maintained total — never directly settable. Every stock change — restock/adjustment/waste/transfer/**sale** — writes an immutable `stockMovements` ledger entry, and the same Admin-SDK write path atomically increments `productStock.quantity` in the same transaction. No edit form ever sets quantity directly.

**Services are walk-in, not appointment-based** — confirmed during Phase 2: no scheduling, calendar, or slot-picker concept anywhere. A service is sold and rendered in one motion, same as a product, just without a stock count behind it.

**Payments** — cash, MTN Mobile Money, and Orange Money, confirmed for Phase 3. A sale can split across any combination of the three. This is manual recording only, by design — no payment-gateway or mobile-money API integration of any kind, no webhook, no balance/transaction verification against either provider. The cashier checks payment on their own phone, outside the app entirely, and just tells the system which method was used; the app's only job is to record that claim. Mobile-money lines (either provider) carry an optional reference/confirmation code the cashier can type in for later reconciliation; cash lines don't. Tax is a stubbed `taxAmount` field defaulting to 0 until jurisdiction is locked — don't build real tax-rate logic yet.

**Voiding a sale never rewrites it** — a void adds `voidedAt`/`voidedBy`/`voidReason` to the existing sale doc; the original line items, prices, and payments are never altered or removed. Reversal is ledger-driven, not re-derived from `lineItems`: it queries the original sale's actual `stockMovements` entries (by `saleId` + `type: 'sale'`) and negates each one, rather than re-reading the sale's line items and recomputing what should be reversed — undoing exactly what the ledger recorded as having happened, not re-deriving intent from a separate structure. Each reversal writes a new `stockMovements` entry (type `void`, positive delta, same `saleId` link) and atomically increments `productStock` back up, in the same transaction as the sale-doc update. Only `branch_manager` (their own branch only), `admin`, and `super_admin` can void — not `cashier`, deliberately: the same reasoning as the strict login path applies here, the person best positioned to abuse a control shouldn't be the one holding it unsupervised. A sale can be voided exactly once — the transaction checks `voidedAt` is unset before proceeding. Whenever revenue reporting gets built, voided sales need to be excluded from totals; they stay in the collection for the audit trail, not folded into anything financial.

**Customers are org-wide, like products** — a customer is the same person everywhere, not owned by whichever branch first registered them; only individual sales carry which branch a purchase actually happened at. Phone number is the unique identifier — enforce that with a real transaction-wrapped check, the same fix already applied to the SKU race condition in Phase 2, not a plain read-then-write repeating that mistake. Linking a customer to a sale is optional, not required — walk-in/anonymous sales work exactly as they did in Phase 3; CRM sits on top of checkout, it doesn't gate it.
