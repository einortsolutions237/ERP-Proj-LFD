# Phase 30.1 — Lab Scan Uploads Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let `lab_staff`/`doctor` attach one or more scanned pages to an existing lab result, and let anyone with lab-view access retrieve them — wiring Phase 30's generic foundation into the clinical UI for the first time, plus wrapping the known Storage-error gap so a real user sees a clear message instead of a blank failure.

**Architecture:** No changes to Phase 30's foundation files. `getLabRecords.ts` (the existing data-fetch function `LabSection.tsx` already consumes) is extended to also return each result's own document ID and its list of existing attachments — reusing that function's existing single-audited-call-site rather than adding a new listing route. A new small client component (`AttachScanForm.tsx`) POSTs to the existing `POST /api/attachments`; existing attachments render as plain download links (`GET /api/attachments/[id]`, no viewer/preview). Both `/api/attachments` routes get one narrowly-scoped addition each: the Storage SDK call is wrapped in `try/catch`, returning a clear `502` JSON error instead of an uncaught `500` — everything else in both routes (capability gating, validation order, the existence check) stays byte-identical.

**Tech Stack:** Next.js App Router (one new `'use client'` component, one Server-Component data-layer extension), Firebase Admin SDK (Firestore only — no new Storage interaction beyond what Phase 30 already built), Vitest against the Firestore/Auth/Storage emulators.

## Global Constraints

- No new capability anywhere — reuse `clinical.lab.results.enter` (upload) and `clinical.lab.view` (retrieve/list) exactly as they already gate the rest of the lab UI.
- Zero changes to `src/lib/attachments/capabilityMap.ts` or `storage.rules`.
- The two `/api/attachments` routes get exactly one change each — a `try/catch` around the Storage SDK call, returning a clear JSON `502`. No other line in either route changes: capability checks, validation order, the file-type/size checks, the related-doc existence check, and the Firestore write/audit-log calls are all untouched.
- Multiple attachments per lab result must work — no artificial one-attachment limit anywhere.
- No document viewer/preview — a plain download link (`<a href="/api/attachments/{id}">`) is the entire "view" affordance.
- No new Firestore composite index.

## Investigation findings (read before touching anything)

1. **A scan cannot be attached in the same request as result creation.** `POST /api/attachments` requires `relatedDocId` to reference an *already-existing* document (`src/app/api/attachments/route.ts:50-54`, unchanged by this phase) — but a lab result doesn't exist until `POST /api/lab-results` (in `LabResultForm.tsx`'s own submit) has already succeeded. **Decision:** the attach-scan control goes in `LabSection.tsx`, where results are already displayed after creation — not into `LabResultForm.tsx`'s creation submit. The brief's own "(or wherever results are entered)" phrasing explicitly allows this, and it's the more useful behavior anyway: a scan can be attached to *any* existing result, not only ones created in the current session.
2. **`getLabRecords.ts`'s returned `LabOrderRow.result` currently omits the lab result's own document ID entirely** (`src/lib/clinical/getLabRecords.ts:70-83`) — only its data fields. Both uploading (needs `relatedDocId`) and listing existing attachments need this ID. This file is a Phase 15/19.2 clinical file, not a Phase 30 foundation file — extending it is in scope and necessary.
3. **There is no "list attachments for a relatedDocId" route** — only single-attachment `GET /api/attachments/[id]`. Rather than adding a new route to the Phase 30 foundation (out of this phase's scope per the brief), the attachment list for each result is fetched *inside* `getLabRecords.ts` itself, reusing its existing single-call-site audit pattern (the function already writes one `lab_view` audit entry per call — attaching the attachment list to the same call adds no new audit surface, and no new capability check, since `clinical.lab.view` already gates the whole function).
4. **No new composite index**: the attachments-per-result query (`where('relatedCollection','==','labResults').where('relatedDocId','==',resultId)`) deliberately has no `.orderBy()` — sorting by `createdAt` happens in memory after the fetch instead. An `orderBy` on a field outside the equality filters would need a new Firestore composite index, exactly the recurring bug class this project's own history has hit before (Phase 18, Phase 19.2) — avoided here since each result's attachment list is always short.
5. **Reconciling "zero changes to the routes' core logic" with the explicit request to wrap Storage errors**: the brief names these as two separate instructions, and they only look contradictory. "Core logic" means capability gating, validation ordering, and the existence check — none of that changes. The Storage-error wrap is the one explicitly-authorized, narrowly-scoped exception: a `try/catch` around exactly the Storage SDK call (`bucket.file(...).save(...)` in `POST`, `bucket.file(...).download()` in `GET`), nothing else in either route touched. The brief names `POST` specifically ("the first real UI a user would see that error in"); this plan also wraps `GET`'s Storage call for the identical reason (same failure class, equally trivial fix) — flagging this small, deliberate scope extension explicitly rather than doing it silently.
6. **Testing the wrap**: the Storage *emulator* itself works normally — only the real production bucket is missing — so a failure has to be simulated. The test spies on the real `getAdminStorage` export (`vi.spyOn` on the namespace import, `mockImplementationOnce` to throw) for exactly one call, then restores it. This is boundary-failure-injection on an external SDK call, not mocking this app's own business logic — consistent with this project's test-hygiene standard, which discourages mocking domain logic, not mocking a third-party client specifically to exercise an otherwise-untriggerable error branch. If `vi.spyOn` on the plain namespace import doesn't intercept the call in this Vitest setup (an environment-specific risk this project has hit with other tooling before), fall back to `vi.mock('@/lib/firebase/admin', async (importOriginal) => { const actual = await importOriginal(); return { ...actual, getAdminStorage: vi.fn(actual.getAdminStorage) } })` at the top of the test file and `vi.mocked(getAdminStorage).mockImplementationOnce(...)` in the specific test — report which approach was needed.

```typescript
import { getAdminFirestore } from '@/lib/firebase/admin'
import { writeAuditLog } from '@/lib/audit/log'
import { hasCapability } from '@/lib/auth/permissions'
import { AuthError, type SessionUser } from '@/lib/auth/server-guard'
import type { LabOrder, LabOrderStatus } from '@/lib/types/labOrder'
import type { LabResult, LabResultFlag } from '@/lib/types/labResult'
import type { Attachment } from '@/lib/types/attachment'

export interface LabResultValueRow {
  parameter: string
  value: string
  unit: string | null
  referenceRange: string | null
  flag: LabResultFlag | null
}

export interface LabResultAttachmentRow {
  id: string
  fileName: string
  mimeType: string
  sizeBytes: number
  createdAt: string
}

export interface LabOrderRow {
  id: string
  customerId: string
  doctorUid: string
  doctorName: string
  testName: string
  instructions: string | null
  status: LabOrderStatus
  orderedAt: string
  result: {
    id: string
    values: LabResultValueRow[]
    notes: string | null
    enteredBy: string
    enteredByName: string
    enteredAt: string
    attachments: LabResultAttachmentRow[]
  } | null
}

// Called by both GET /api/lab-orders and the customer detail page's Lab
// section (a Server Component, same direct-in-process pattern as
// getPatientTreatments/getAppointments) — same single-call-site-for-
// audit-logging discipline, so "viewing lab data is read-audit-logged"
// is true by construction. Re-checks the capability itself rather than
// trusting the caller already did, same belt-and-suspenders discipline
// as its two clinical precedents.
//
// Phase 30.1: also resolves each result's own attachments (Phase 30's
// generic attachments collection, filtered to relatedCollection ===
// 'labResults') at this same call site, rather than adding a separate
// listing route — no new audit surface, no new capability check, since
// clinical.lab.view already gates the whole function.
export async function getLabRecords(customerId: string, viewer: SessionUser): Promise<LabOrderRow[]> {
  if (!hasCapability(viewer.role, 'clinical.lab.view')) {
    throw new AuthError('Forbidden', 403)
  }

  const db = getAdminFirestore()
  const ordersSnap = await db
    .collection('labOrders')
    .where('customerId', '==', customerId)
    .orderBy('orderedAt', 'desc')
    .get()

  const orders = ordersSnap.docs.map((d) => ({ id: d.id, data: d.data() as LabOrder }))
  const uniqueDoctorUids = Array.from(new Set(orders.map((o) => o.data.doctorUid)))
  const doctorDocs = await Promise.all(uniqueDoctorUids.map((uid) => db.collection('staff').doc(uid).get()))
  const doctorNames: Record<string, string> = {}
  uniqueDoctorUids.forEach((uid, i) => {
    doctorNames[uid] = (doctorDocs[i].data()?.name as string | undefined) ?? uid
  })

  // At most one result per order (a labOrder <-> labResult relationship
  // is 1:0-or-1, not 1:many) — a single equality-filtered, limit(1) query
  // per order, same Promise.all fan-out shape as the name lookups above.
  const resultSnaps = await Promise.all(
    orders.map((o) => db.collection('labResults').where('labOrderId', '==', o.id).limit(1).get())
  )
  const uniqueEnteredByUids = Array.from(
    new Set(resultSnaps.flatMap((s) => s.docs.map((d) => (d.data() as LabResult).enteredBy)))
  )
  const enteredByDocs = await Promise.all(uniqueEnteredByUids.map((uid) => db.collection('staff').doc(uid).get()))
  const enteredByNames: Record<string, string> = {}
  uniqueEnteredByUids.forEach((uid, i) => {
    enteredByNames[uid] = (enteredByDocs[i].data()?.name as string | undefined) ?? uid
  })

  // Attachments per result — sorted in memory (not via .orderBy) to avoid
  // needing a new Firestore composite index for what's always a short list.
  const resultIds = resultSnaps.map((s) => s.docs[0]?.id).filter((id): id is string => id !== undefined)
  const attachmentSnaps = await Promise.all(
    resultIds.map((resultId) =>
      db
        .collection('attachments')
        .where('relatedCollection', '==', 'labResults')
        .where('relatedDocId', '==', resultId)
        .get()
    )
  )
  const attachmentsByResultId: Record<string, LabResultAttachmentRow[]> = {}
  resultIds.forEach((resultId, i) => {
    attachmentsByResultId[resultId] = attachmentSnaps[i].docs
      .map((d) => {
        const a = d.data() as Attachment
        return {
          id: d.id,
          fileName: a.fileName,
          mimeType: a.mimeType,
          sizeBytes: a.sizeBytes,
          createdAt: a.createdAt.toDate().toISOString(),
        }
      })
      .sort((a, b) => a.createdAt.localeCompare(b.createdAt))
  })

  const rows: LabOrderRow[] = orders.map(({ id, data }, i) => {
    const resultDoc = resultSnaps[i].docs[0]
    const result = resultDoc
      ? (() => {
          const r = resultDoc.data() as LabResult
          return {
            id: resultDoc.id,
            values: r.values,
            notes: r.notes,
            enteredBy: r.enteredBy,
            enteredByName: enteredByNames[r.enteredBy] ?? r.enteredBy,
            enteredAt: r.enteredAt.toDate().toISOString(),
            attachments: attachmentsByResultId[resultDoc.id] ?? [],
          }
        })()
      : null

    return {
      id,
      customerId: data.customerId,
      doctorUid: data.doctorUid,
      doctorName: doctorNames[data.doctorUid] ?? data.doctorUid,
      testName: data.testName,
      instructions: data.instructions,
      status: data.status,
      orderedAt: data.orderedAt.toDate().toISOString(),
      result,
    }
  })

  await writeAuditLog({
    action: 'lab_view',
    actorUid: viewer.uid,
    actorEmail: viewer.email,
    targetUid: customerId,
    branchId: null,
    details: null,
  })

  return rows
}
```

```typescript
'use client'
import { useState, useRef } from 'react'

export interface AttachScanFormProps {
  labResultId: string
  onDone: () => void
}

export default function AttachScanForm({ labResultId, onDone }: AttachScanFormProps) {
  const [uploading, setUploading] = useState(false)
  const [error, setError] = useState<string | null>(null)
  const fileInputRef = useRef<HTMLInputElement>(null)

  async function handleUpload(e: React.FormEvent) {
    e.preventDefault()
    const file = fileInputRef.current?.files?.[0]
    if (!file) {
      setError('Choose a file first')
      return
    }
    setError(null)
    setUploading(true)

    const formData = new FormData()
    formData.set('relatedCollection', 'labResults')
    formData.set('relatedDocId', labResultId)
    formData.set('file', file)

    try {
      const res = await fetch('/api/attachments', { method: 'POST', body: formData })
      const body = await res.json()
      if (!res.ok) {
        setError(body.error ?? 'Upload failed')
        setUploading(false)
        return
      }
      if (fileInputRef.current) fileInputRef.current.value = ''
      setUploading(false)
      onDone()
    } catch {
      setError('Upload failed')
      setUploading(false)
    }
  }

  return (
    <form onSubmit={handleUpload} className="flex flex-wrap items-center gap-2">
      <input
        ref={fileInputRef}
        type="file"
        accept="image/jpeg,image/png,application/pdf"
        className="text-xs text-ink"
      />
      <button
        type="submit"
        disabled={uploading}
        className="shrink-0 rounded-md border border-mist px-2 py-1 text-xs text-ink transition-colors hover:bg-mist/40 disabled:opacity-50"
      >
        {uploading ? 'Uploading…' : 'Attach scan'}
      </button>
      {error && <p className="w-full text-xs text-danger">{error}</p>}
    </form>
  )
}
```

Note on the explicit `setUploading(false)` on the success path (unlike `LabResultForm.tsx`'s precedent, which only resets its equivalent flag on the error path): `LabResultForm` unmounts on success (its parent removes it via a toggle). `AttachScanForm` stays permanently mounted next to each result — `onDone`'s `router.refresh()` re-fetches server data but does not unmount this client component, so skipping the reset here would leave the button stuck on "Uploading…" forever after every successful upload. This is a deliberate, necessary divergence from the sibling forms' style, not an inconsistency to "fix" later.
