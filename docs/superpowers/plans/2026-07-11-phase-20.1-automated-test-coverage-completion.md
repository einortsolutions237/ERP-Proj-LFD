## Environment work required before any test could run

This machine had no prior test infrastructure and needed real setup, not just code:

1. **No JRE.** The Firestore emulator is Java-based; none was installed. Installed Eclipse Temurin 21 via `winget`, user-approved before installing.
2. **gRPC connection resets.** The Firestore emulator's default transport (gRPC) connected, then got reset by the emulator's Netty server ~13 seconds in on every write — a Windows/grpc-js/Java-emulator interop issue, confirmed via `firestore-debug.log` (`java.net.SocketException: Connection reset`), not an application bug. Fixed via `db.settings({ preferRest: true })` (Firebase's own documented workaround for this exact problem), applied once per test file in `tests/setup/env.ts` using the existing exported `getAdminFirestore()` — zero changes to `src/lib/firebase/admin.ts`.
3. **REST transport loses the Admin SDK's automatic rules bypass.** Over gRPC, the Admin SDK auto-attaches a `Bearer owner` header that skips `firestore.rules` entirely; over REST, this doesn't fire — confirmed empirically, including that even a credential-free connection still tried real Google OAuth and failed. Our dummy `cert()` credential produced a token the emulator evaluated against the real (default-deny) `firestore.rules`, returning 403 on every write. Fixed with a new **test-only** `firebase.testing.json` (emulator ports + a pointer to `firestore.test.rules`, a permissive `allow read, write: if true` file used only by the emulator) — referenced solely via `--config firebase.testing.json` in the `test`/`test:watch` npm scripts. The real `firebase.json` and the real `firestore.rules` used for production deployment are completely untouched (confirmed via empty `git diff` throughout). This phase does not test Firestore rules — only application-layer logic via the Admin SDK — so permissive emulator-only rules don't undermine anything this phase verifies.

All three were diagnosed with real evidence (debug logs, isolated reproduction scripts, empirical A/B tests of each hypothesis) before any fix was proposed, and the fix was presented to and approved by the user before being built, since it touched two assumptions baked into the original plan (transport mode, and "Admin SDK always bypasses rules").

## What each suite verifies

**Role×capability matrix (Priority 1).** An exhaustive snapshot over every role × every capability (14 roles × 41 capabilities = 574 cells), plus 10 named structural-invariant assertions: `super_admin` holds everything, `clinical.record.create` is exactly `[super_admin, doctor]`, `general_manager`'s three deliberate asymmetries hold (no `crm.customer.view`, no clinical authoring, no seminar-attendance recording), `BRANCH_LOCKED_ROLES` is exactly `[branch_manager, cashier, inventory_manager]`, `admin`'s exact capability footprint, and the `lab_staff`/`nurse` capability splits. This is the test that would have caught the Phase 13-14 `admin` clinical-access gap.

*Correction made mid-implementation:* the plan's own draft assertions for `admin`'s capability count and clinical-capability exclusion were wrong — they missed that `hr.leave.request`/`hr.attendance.self`/`messaging.access` are granted to every role via `ALL_ROLES` (entirely outside the Phase 17 restructuring), and that `clinical.questionnaire.manage` is a deliberate Phase 19.1 exception explicitly granted to `admin`. The implementer correctly escalated rather than silently editing the test to pass; the controller independently re-traced `ROLE_CAPABILITIES` by hand, confirmed the real 9-capability set, and supplied corrected assertions. The plan document has been corrected in place with an explanatory comment. **The application code (`src/lib/auth/permissions.ts`) was never wrong — only the plan's test assertions were.**

**`canMessage` truth table (Priority 2).** Exhaustive over all 14×14 ordered role pairs × same/different branch (392 generated cases) plus 5 named cases (self-message, `branch_manager`↔`branch_manager`, `general_manager`→staff, cross-branch staff→`branch_manager`, `it_admin`'s flat line both directions). The reviewer confirmed the suite's symmetry checks are genuinely load-bearing (a real asymmetry bug would fail at least one ordered-pair assertion, not merely re-check a symmetric oracle by coincidence) and that the self-message guard is correctly tested as a separate named case since the generated matrix structurally can't exercise it (always uses distinct uids).

**Sale transaction (Priority 3).** All 8 named behaviors from the plan: server-side price resolution (client-supplied price ignored, verified against the *persisted* document, not just the response), correct stock decrement with `resultingQuantity` cross-checked against an independently-read `productStock.quantity`, payment-sum validation straddling the real `EPSILON = 0.01` boundary, sufficient-stock sale with no backorder, insufficient-stock sale landing at exactly zero stock with a correctly-sized `pendingDeliveries` record, backorder-without-customer rejecting 409 with nothing written (verified via before/after document counts, not just the status code), and idempotency-key replay returning the existing sale with the Firestore document count unchanged. The Opus reviewer specifically ruled out the "test passes even if the underlying logic is broken" failure mode for every one of the 8 tests — each reads real emulator-backed state rather than trusting a mock or a response body alone. `src/app/api/sales/route.ts` confirmed genuinely untouched throughout.

**`isBranchLocked` read scoping (Priority 4).** `GET /api/sales`, `/api/staff`, `/api/departments`, `/api/stock` — branch-locked roles get only their branch, org-wide roles get everything, verified against real `branchId` fields in response payloads (not row counts). *Correction made mid-implementation:* the plan's fixture seeded every test staff account in Branch A only, leaving the staff test's "org-wide sees both branches" assertion structurally unpassable regardless of route correctness. The implementer ran the suite exactly as planned first, observed the expected failure, correctly diagnosed it as a fixture gap (not an app bug, confirmed by reading the route and role-list constants directly), and fixed it with one additive seed call. **One honest limitation on record:** `GET /api/staff`'s current capability holders (`super_admin`/`general_manager`/`hr_admin`, none branch-locked) mean that test's regression coverage is narrower than its three siblings — it can't fail via "a branch-locked role wrongly granted this capability" since no such role exists today, but it does still guard against a future accidental role-list change or a regression in the route's own `isBranchLocked` branch.

**Clinical wall (Priority 5).** `nurse` can record intake but is blocked from authoring treatments; `lab_staff` can enter results but is blocked from ordering; `doctor` retains both ordering and treatment authorship; `cashier` is blocked from all five clinical routes tested. **One real cross-file infrastructure issue found and fixed:** Vitest schedules test files with some concurrency against a single shared emulator instance (not fully isolated per file), so this suite's literal `cashier@test.local` collided with the sale-transaction suite's account of the same email — the Auth emulator rejects duplicate email creation. Fixed by renaming this file's four fixture emails with a `-cw` suffix; no other committed file was touched (confirmed via empty diff against `sales.test.ts`/`branch-scoping.test.ts`). **This is a structural risk for any future test file that reuses a plain literal email** (e.g. `doctor@test.local`) — worth a naming convention (a suffix per file, or a shared counter) rather than rediscovering the collision each time a new integration suite is added. No fix was made to `tests/setup/fixtures.ts` itself in this phase, since the concrete instance was resolved and nothing else currently collides — flagged here for whoever adds the next integration test file.
