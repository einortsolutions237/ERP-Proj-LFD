# Phase 19.1 (Nurse & Patient Intake) — Completion Report

**Date:** 2026-07-07

## Summary

Added a `nurse` role and the patient-intake workflow that runs between a booked appointment and the doctor's consultation: one-time demographic facts (upsert, one record per customer), a shared editable questionnaire template, and per-visit vitals/answers where each answer's question text is a server-built snapshot of the template at the moment the visit was recorded — never a live reference. Three new, deliberately standalone capabilities (`clinical.intake.record`, `clinical.questionnaire.manage`, `clinical.intake.view`) were added without touching the two pre-existing clinical-access constants (`CLINICAL_ROLES`, `CLINICAL_VIEW_ROLES`), so `nurse` is structurally walled off from diagnosis/treatment/appointment/lab access, not just correctly configured today.

## What shipped

- `src/lib/auth/permissions.ts` — `nurse` added to `ROLES`; three new capabilities, each backed by its own standalone role list (`INTAKE_RECORD_ROLES = ['super_admin', 'doctor', 'nurse']`, `QUESTIONNAIRE_MANAGE_ROLES = ['super_admin', 'general_manager', 'admin', 'nurse']`, `INTAKE_VIEW_ROLES = ['super_admin', 'doctor', 'medical_secretary', 'general_manager', 'nurse']`) — `CLINICAL_ROLES`/`CLINICAL_VIEW_ROLES` untouched. `nurse` is absent from `BRANCH_LOCKED_ROLES` (clinical-adjacent data stays org-wide) and `STRICT_AUDIT_ROLES` (client-SDK login path).
- Three new, fully-closed Firestore collections: `patientDemographics` (doc ID = `customerId`, upserted), `intakeQuestionnaire` (single doc, ID `'default'`), `nursingVisits` (one doc per visit, composite index on `customerId` + `recordedAt`).
- `src/lib/clinical/getPatientIntake.ts` — shared read helper (customer page + API route), single `intake_view` audit call site, mirroring `getPatientTreatments`/`getLabRecords` exactly.
- `GET`/`POST /api/patient-intake*` — combined read, demographics upsert, and visit creation. The visit route reads `intakeQuestionnaire/default` fresh at write time and zips it positionally with client-submitted answers — the `question` field can never come from client-supplied text.
- `GET`/`PUT /api/intake-questionnaire` — the shared template; read is not audit-logged, edit is (`intake_questionnaire_edit`).
- A new "Patient demographics" / "Nursing visits" section on the customer detail page, with `clinical.intake.view` added to that page's top-level `requireAnyCapability` guard — the exact class of gap Phase 16 shipped once already (a built-but-unreachable section) — confirmed not repeated, independently, at both the task-review and final-review stage.
- **Added post-final-review, per explicit user decision:** a questionnaire editor page (`/intake-questionnaire`) and `QuestionnaireEditor.tsx`, plus a Sidebar nav entry gated on `clinical.questionnaire.manage`. The original plan had an internal inconsistency — Task 5 said the `PUT` route would be "consumed by Task 6's questionnaire-editor UI," but Task 6 never actually included one — leaving the capability fully wired with no way to exercise it except a direct API call. Closed as Task 8.
- A small bug found during the final review and fixed directly: `NursingVisitForm`'s questionnaire fetch had no `.catch`, so a network failure left the form stuck on "Loading questionnaire…" forever with no error shown.

## Review

Six task-level reviews (Opus on Tasks 1 and 4 — the capability grant and the write/snapshot routes; Sonnet on the rest) plus a final whole-branch review (Opus) plus one additional Sonnet review for the post-review Task 8 addition. **Zero Critical findings across every review.** The one substantive Important finding from the final review (the missing questionnaire editor) was resolved by adding it as Task 8, per explicit user decision, rather than left as an unconfirmed assumption. A handful of Minor findings were triaged and left as-is (index-as-key in list renders, a redundant inner try/catch, an unchecked doc-shape cast on the questionnaire template) — none rose above cosmetic, and none were introduced by any implementer deviation from the plan's own code.

One documented implementer deviation, independently confirmed cosmetic by both the Task 4 reviewer and the final whole-branch reviewer: the demographics route's write payload had its explicit `PatientDemographics` type annotation removed (kept as an unannotated object) because the plan's sample code didn't type-check against a `Date`/`Timestamp` union assigned to a strictly `Timestamp`-typed field — matching an existing precedent elsewhere in this codebase (`seminar-attendance/route.ts`). No field, value, or logic changed.

Lint was run proactively across every file this phase touched before the final review (a lesson carried over from Phase 19, where lint caught a real bug the diff-only review had missed) — zero issues.

## Outstanding / not in this phase

- Flagged Gaps #1–#2 from the plan remain: `nurse` has no in-app path to discover which customer to intake (no `crm.customer.view`, no appointment-schedule visibility — matching the `protocol` precedent from Phase 16), and `nursingVisits.appointmentId` has API-level support but no UI picker.
- Flagged Gap #3 (positional question/answer alignment) has a narrow, inherent race noted by the final review: if the template is edited between a nurse loading the visit form and submitting it, answers could misalign against a since-reordered question list. Not fixed this phase — the underlying design (server-side snapshot at write time) is correct and was the one property under scrutiny; this is a separate, narrower hardening for a future phase if template edits become frequent enough to matter.
- Phase 19.2 (`lab_staff`) was not built — it appeared only in the stale pasted CLAUDE.md draft, never in an actual request this session.
