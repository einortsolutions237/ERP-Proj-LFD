# Phase 13.1 — Medical Secretary Role Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task.

**Goal:** Add a `medical_secretary` role with org-wide clinical read access and commercial (purchase-history) read access, but no authoring capability on either side of the wall. Reserve two future capabilities (`clinical.appointments.manage`, `seminars.attendance.view`), granted to nobody yet.

**Architecture:** This phase is entirely contained in `src/lib/auth/permissions.ts` — no new collections, no new routes, no new UI. Every downstream behavior (the customer detail page's `canViewCommercial`/`canViewClinical`/`canCreateTreatment` gates, the staff-creation role dropdown, the roles/permissions matrix) is already capability-driven from Phase 13, so wiring the role correctly into `ROLE_CAPABILITIES` is sufficient — nothing else needs to change or even be touched.

## Global Constraints

- No appointments or seminars UI/logic — the two reserved capabilities
  exist only in the type system and the (empty) role-grant table. Building
  toward them is explicitly Phase 14/16's job, not this one.
- `medical_secretary` must not gain `clinical.record.create` or
  `crm.customer.manage` — verify by direct API attempt, not just UI
  absence.
- `medical_secretary`'s clinical reads must be exactly as org-wide as
  `doctor`'s — no `isBranchLocked`/`BRANCH_LOCKED_ROLES` involvement at all.
- `medical_secretary` is not `STRICT_AUDIT_ROLES` (client-SDK login path).
- TD-3 (customer deletion not checking `treatments`) must not be touched
  — this phase has no reason to open `api/customers/[id]/route.ts`'s
  `DELETE` handler, and shouldn't.

## The one judgment call, flagging before implementation

`crm.customer.view` and `clinical.record.view` currently share role-group
constants (`CASHIER_BRANCH_MGR` and `CLINICAL_ROLES` respectively) with
capabilities `medical_secretary` must **not** hold (`crm.customer.create`,
`pos.sale.create`, `pos.sale.view` share `CASHIER_BRANCH_MGR`;
`clinical.record.create` shares `CLINICAL_ROLES`). Adding
`medical_secretary` to either shared constant directly would silently
grant it `pos.sale.create`/`.view` or `clinical.record.create` too —
exactly the "no accidental inheritance" failure mode Phase 13 was
careful about. The only clean fix: split each shared constant into a
`*_VIEW`-specific one carrying the new role, leaving a `*_CREATE`
(or equivalently-named) constant with the original role list, unchanged,
for the capabilities that must not gain `medical_secretary`. This touches
the wiring for four existing capabilities
(`pos.sale.create`/`pos.sale.view`/`crm.customer.create` via the
`CASHIER_BRANCH_MGR` split, `clinical.record.create` via the
`CLINICAL_ROLES` split) even though none of their actual granted roles
change — flagging this because the reviewer needs to verify it's a pure
refactor for those four, not a behavior change.

## Files — single task (this entire phase is one cohesive permissions change)

1. `src/lib/auth/permissions.ts` — Modify:

   Add `'medical_secretary'` to `ROLES`:
   ```ts
   export const ROLES = [
     'super_admin', 'admin', 'branch_manager', 'hr_admin', 'finance_admin', 'it_admin', 'cashier', 'doctor', 'medical_secretary',
   ] as const
   ```

   Add `'seminars'` to `MODULES` (needed for `seminars.attendance.view`'s
   module-prefix naming convention — `clinical.appointments.manage`
   reuses the existing `'clinical'` module, no change needed there):
   ```ts
   export const MODULES = ['admin', 'pos', 'inventory', 'crm', 'accounting', 'hr', 'reporting', 'clinical', 'seminars'] as const
   ```

   Add two capabilities to the `Capability` union:
   ```ts
     | 'clinical.appointments.manage'
     | 'seminars.attendance.view'
   ```

   Add both to `CAPABILITY_MODULE`:
   ```ts
     'clinical.appointments.manage': 'clinical',
     'seminars.attendance.view': 'seminars',
   ```

   Split `CASHIER_BRANCH_MGR` — keep it exactly as-is for
   `pos.sale.create`/`pos.sale.view`/`crm.customer.create`, and add a new
   constant for `crm.customer.view` specifically:
   ```ts
   const CRM_VIEW_ROLES: RoleId[] = ['super_admin', 'admin', 'branch_manager', 'cashier', 'medical_secretary']
   ```
   Change `ROLE_CAPABILITIES['crm.customer.view']` from `CASHIER_BRANCH_MGR`
   to `CRM_VIEW_ROLES`. Do NOT change `crm.customer.create`,
   `pos.sale.create`, or `pos.sale.view` — they keep `CASHIER_BRANCH_MGR`
   exactly as today.

   Split `CLINICAL_ROLES` similarly — keep it exactly as-is for
   `clinical.record.create`, and add a new constant for
   `clinical.record.view` specifically:
   ```ts
   const CLINICAL_VIEW_ROLES: RoleId[] = ['super_admin', 'admin', 'doctor', 'medical_secretary']
   ```
   Change `ROLE_CAPABILITIES['clinical.record.view']` from
   `CLINICAL_ROLES` to `CLINICAL_VIEW_ROLES`. Do NOT change
   `clinical.record.create` — it keeps `CLINICAL_ROLES` exactly as today
   (still `['super_admin', 'admin', 'doctor']`, unaffected).

   Add the two reserved capabilities to `ROLE_CAPABILITIES`, granted to
   nobody:
   ```ts
     'clinical.appointments.manage': [],
     'seminars.attendance.view': [],
   ```

   **Do NOT touch:** `STRICT_AUDIT_ROLES` (no `medical_secretary`),
   `BRANCH_LOCKED_ROLES` (no `medical_secretary`), `ALL_ROLES` (it's a
   spread of `ROLES`, so `medical_secretary` automatically gets
   `hr.leave.request`/`hr.attendance.self` with no further edit — same
   mechanism as every prior role addition), `APPROVER_ROLES`,
   `REPORTS_ROLES`, `ADMIN_HR`, `ADMIN_ONLY`, `ADMIN_BRANCH_MGR`,
   `ADMIN_IT`, or any other existing `ROLE_CAPABILITIES` entry not named
   above. `hasCapability`/`isSuperAdmin`/`isBranchLocked` function bodies
   are unchanged.
