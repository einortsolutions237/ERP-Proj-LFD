## Summary

The second half of offline-capable POS. Checkout now works with no network connectivity: a completed sale writes to an IndexedDB queue (client-generated, never-regenerated idempotency key) instead of calling `POST /api/sales`, showing an immediate provisional receipt. A sync engine replays queued sales sequentially once connectivity returns, distinguishing a genuine network failure (retry later, don't touch the rest of the queue) from a real server rejection (mark `needs_attention`, keep going — the backorder-without-customer 409 is the case that matters most). `api/sales/route.ts` gained an optional `clientIdempotencyKey` check — its only change — as the second deliberate touch to that transaction (the first was Phase 18), preventing a double-sale if a sync response is lost after the server already committed. A queue status indicator with a needs-attention resolution flow is wired into the nav shell for `cashier`/`branch_manager` only.

- **Resolution flow**: attached a real customer via the panel's lazy `GET /api/customers` fetch, retried, and confirmed success — a `sales` doc with the correct `customerId` and a `pendingDeliveries` doc (`quantityOwed: 2`, `status: 'pending'`) were both created correctly, stock confirmed still at exactly 0 (never negative).

- **Catalog/stock usable offline**: the offline banner rendered with a real "last sync" timestamp, and cart/search continued working with zero network calls while offline.

## Scope boundary held

No multi-device conflict resolution, no background sync while the tab is closed, no service worker / hard-offline-reload support — all explicitly out of scope per the plan's Global Constraints, none attempted.

## Decisions (all 7, approved as written, no objections)

1. Idempotency key generated once and sent on the first live attempt too, not just queued retries.
2. Network-level failure stops the sync pass and resets to `queued`; a real HTTP rejection marks only that item `needs_attention` and continues.
3. Hand-written IndexedDB wrapper, no new npm dependency.
4. Catalog caching is write-through within the current tab session only — no service worker.
5. Queue visibility gated to `cashier`/`branch_manager`.
6. Needs-attention resolution fetches customers from the existing `GET /api/customers`, on demand.
7. Synced entries auto-clear after a 60-second grace period.
