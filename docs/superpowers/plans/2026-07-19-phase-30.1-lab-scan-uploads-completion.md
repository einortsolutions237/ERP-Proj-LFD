## A real bug found and fixed mid-phase, not part of the original plan

Task 3's implementer discovered that `npm run build` genuinely fails on `main` — including the already-tagged `phase-30-baseline` — due to a real TypeScript error: `return new Response(buffer, {...})` in the retrieval route, where `Buffer<ArrayBufferLike>` isn't assignable to the `BodyInit` type `Response` expects. This has been latent since Phase 30 shipped, because neither of that phase's two tasks ever ran a build check, only `npm test`. The implementer's first fix (`buffer as any`) was correctly flagged Important by the task reviewer on two independent dimensions — an out-of-scope file touch, and a type-safety-defeating workaround on a security-relevant, previously Opus-reviewed route — and was corrected to `new Response(new Uint8Array(buffer), {...})`, verified by the re-reviewer (and independently again by the final whole-branch reviewer) to resolve `npx tsc --noEmit` with zero errors project-wide. The final review judged this an acceptable, transparent resolution: the fix is the *only* out-of-scope touch anywhere in the diff, lands honestly in Task 3's own commit (not smuggled into Task 2's), and was necessary to make the branch buildable at all.

## A second, systemic test-infrastructure fix applied at the finishing stage

The Task 3 re-reviewer observed `tests/integration/getLabRecords.test.ts` (Task 1's new file) failing intermittently under the full suite. The final whole-branch reviewer went further and reproduced it **deterministically**: 4/4 passing in isolation, 3/4 failing when scheduled concurrently with `attachments.test.ts` — confirming the application and test code are both correct, and this is purely this project's already-documented (CLAUDE.md's Phase 23 process note) cross-file emulator-pollution class, sharpened by this phase's new file rather than caused by it. Every integration test file's `beforeAll` calls `resetEmulator()`, an unscoped whole-database wipe — safe run one-at-a-time, unsafe under Vitest's default file-level concurrency. Per the reviewer's recommendation, fixed at the root rather than with another per-file workaround: `vitest.config.ts` now sets `fileParallelism: false`. Verified clean across three full-suite runs afterward (502/502 each), committed separately (`d07e2b9`) since it's test-infrastructure, not phase-scoped feature work.
