## What shipped

Gross-pay-only staff payroll tracking ‚Äî the second half of the Accounting/Payroll priority, directly on Phase 26's expense/P&L foundation. `finance_admin`/`super_admin` record per-staff-member payroll against a standing `baseSalary` reference figure on the staff record; `general_manager`/`hr_admin` get oversight visibility. No batch payroll runs, no net-pay/withholding calculation, no tax logic, no payslip generation anywhere.

- **`baseSalary` on the staff record (`b3056be`, `842213b`)** ‚Äî added to the `Staff` type, editable only via `PATCH /api/staff/[staffId]` (gated `admin.staff.edit` = `super_admin`/`general_manager`/`hr_admin`), deliberately never settable via `POST /api/staff` (gated `admin.staff.create`, a different role set that includes plain `admin` ‚Äî Phase 17 excluded `admin` from staff/salary oversight, and creation must not become a backdoor around that). UI: a new field on the staff edit form, edit-mode only.
- **Data model & capabilities (`b3056be`)** ‚Äî new `payrollRecords` collection (fully closed in Firestore rules), `PayrollRecord` type (`staffId`, `payPeriodStart`, `payPeriodEnd`, `grossAmount`, `branchId`, `recordedBy`, `createdAt`, `notes`). Two new capabilities as **standalone** role-list constants ‚Äî not composed by reference from Phase 26's accounting constants, even though membership currently overlaps: `payroll.record.create` (`super_admin`/`finance_admin` only) and `payroll.record.view` (adds `general_manager`/`hr_admin`, view-only, never create).
- **Payroll record creation (`d0793bd`)** ‚Äî `src/lib/payroll/store.ts`'s `createPayrollRecord` resolves `grossAmount` from an explicit override or falls back to the target staff member's current `baseSalary`, rejecting (400) if neither is available. `branchId` is derived **solely from the target staff member's own branch** ‚Äî never the recording user's, and structurally unrepresentable as a client override (the input type has no `branchId` field at all) ‚Äî a deliberate departure from how `expenses.branchId` works, since a payroll record's branch is never ambiguous once `staffId` is known.
- **UI (`83a0223`, `f61d2c8`)** ‚Äî `/payroll` list page (conditional "Record payroll" link gated on `hasCapability`), `/payroll/new` form with a staff picker that reads the `staff`/`branches` collections directly server-side (name/role/branch only, no salary data) rather than requiring `admin.staff.view` ‚Äî `finance_admin` doesn't hold that capability, mirroring the precedent Phase 26 set for its own branch-picker lookups. Sidebar nav entry.

## Decisions confirmed during planning (see plan's own "Decisions made explicit" section)

- `baseSalary` is edit-only (Decision 1) ‚Äî a brand-new staff member has `baseSalary: null`/`undefined` until an `hr_admin`/`general_manager`/`super_admin` sets it via an edit.
- `payrollRecords.branchId` derivation deviates from a literal reading of "same resolution pattern as expenses" (Decision 2) ‚Äî resolved from the target staff member's doc, not the recording user, since the ambiguity expenses has to resolve doesn't exist once `staffId` is known. Confirmed with the user before implementation.
- The two payroll capabilities map to the existing `'accounting'` module (Decision 3), not a new `'payroll'` module.
- Standalone role-list constants, not composed by reference (Decision 4) ‚Äî the same discipline this project adopted for `nurse`/`lab_staff` after composition-by-reference caused silent drift in earlier phases.
- The staff-picker's direct `staff`/`branches` read (Decision 7) avoids inventing a new capability just to populate a dropdown, mirroring Phase 26's own precedent.
