# Phase 19.2 (Laboratory Staff) ‚Äî Completion Report

**Date:** 2026-07-10

## Summary

Added a `lab_staff` role and split Phase 15's single combined `clinical.lab.manage` capability into two standalone capabilities ‚Äî `clinical.lab.order` (place an order, a clinical decision) and `clinical.lab.results.enter` (record results, a specialist-plus-fallback action) ‚Äî so a lab technician can enter results without gaining the authority to order tests, and a doctor retains both. `clinical.lab.view` gained `lab_staff` and `nurse`, and ‚Äî the load-bearing structural change ‚Äî was rewired off `CLINICAL_VIEW_ROLES` (which also backs `clinical.record.view`/`clinical.appointments.manage`) onto its own standalone `LAB_VIEW_ROLES` list, so `lab_staff`/`nurse` cannot silently inherit treatment/appointment access. `labOrders` gained an optional, nullable, server-validated `treatmentId` (order a test from within a treatment record, or standalone); `labResults` gained an optional `notes` field. A new org-wide, cross-branch pending-orders worklist (`/lab-orders/pending`, gated on `clinical.lab.results.enter`) gives the people who act on orders a single queue. No new collections ‚Äî `labOrders`/`labResults` were reused as-is.

## What shipped

- `src/lib/auth/permissions.ts` ‚Äî `lab_staff` added to `ROLES`. `clinical.lab.manage` removed; three lab capabilities now backed by three standalone lists:
  - `LAB_ORDER_ROLES = ['super_admin', 'doctor']` ‚Üí `clinical.lab.order`
  - `LAB_RESULTS_ENTER_ROLES = ['super_admin', 'doctor', 'lab_staff']` ‚Üí `clinical.lab.results.enter`
  - `LAB_VIEW_ROLES = ['super_admin', 'doctor', 'medical_secretary', 'general_manager', 'lab_staff', 'nurse']` ‚Üí `clinical.lab.view`
  - `lab_staff` is absent from `STRICT_AUDIT_ROLES` (client-SDK login path) and its assignment is branch-locked, while the lab capabilities themselves are org-wide (consistent with `clinical.lab.view` having always been org-wide).
- `src/app/api/lab-orders/route.ts` ‚Äî `POST` now gated on `clinical.lab.order`; accepts an optional `treatmentId`, validated to reference a real `treatments` doc **belonging to the same customer** (else 400); persists `treatmentId: null` when omitted.
- `src/app/api/lab-results/route.ts` ‚Äî `POST` now gated on `clinical.lab.results.enter`; accepts an optional `notes` string, persisted on the `labResults` doc. Results entry remains the one transaction-guarded write (atomically creates the result and flips the order to `completed`, 409 on a second submission).
- `src/lib/clinical/getPendingLabOrders.ts` + `src/app/api/lab-orders/pending/route.ts` + `src/app/(dashboard)/lab-orders/pending/page.tsx` ‚Äî the worklist: `status == 'ordered'`, org-wide, no branch filter, gated on `clinical.lab.results.enter`; each row carries patient name, test, ordering doctor, and ordered date; a dedicated `lab_worklist_view` audit action at the single read call site.
- `firestore.indexes.json` ‚Äî added the `labOrders(status ASC, orderedAt ASC)` composite index the worklist query requires (load-bearing ‚Äî confirmed live below).
- UI ‚Äî `LabSection` split its gates into `canOrder`/`canEnterResults`; `ClinicalSection` gained a per-treatment-row "Order lab test" action (gated on `clinical.lab.order`); `LabSection` renders a result's `notes` as a `Note:` line only when present. The customer detail page's top-level guard added `clinical.lab.view`, so a pure `lab_staff` can reach a patient page and see only the Lab section.

## Honest notes

- **Step 9 / Purchase history:** the task brief's Step 9 lists "purchase history" among the sections `general_manager` should see, but `general_manager` does **not** hold `crm.customer.view` (an explicit Phase-17 asymmetry ‚Äî it oversees the business but was not granted commercial customer-record access), so the Purchase history section does not render for it. This is unchanged by Phase 19.2 and is not a regression; it was confirmed and recorded rather than glossed. Every section `general_manager` genuinely had access to before this phase still renders.
- **Environment, not code:** on the first verification pass, the seven page-render / appointment-`PATCH` checks returned HTTP 500. The cause was a *pre-existing, stale* `next dev` process already bound to port 3000 whose React Server Component worker had crashed (`Jest worker encountered ‚Ä¶ child process exceptions`, `EPIPE` in its log) ‚Äî the API routes still answered correctly, but full server-component page renders failed. After killing the stale process and starting a fresh dev server, all seven passed with **no code change**. The API-level permission checks (403/201/200) executed correctly on both servers; only the RSC page renders were affected. This is recorded the same way Phase 18's report recorded a check that passed for a different reason than intended ‚Äî the "failures" were an environment artifact, not a defect.
