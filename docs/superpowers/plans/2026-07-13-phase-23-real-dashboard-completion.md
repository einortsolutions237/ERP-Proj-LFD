## What shipped

Replaced the placeholder "Welcome / Check In" landing page (`src/app/(dashboard)/dashboard/page.tsx`) with a real, role-aware dashboard — the first genuinely new functionality shipped since Phase 20 (Phases 21/22 were presentation-only design rollouts). Four independently capability-gated widgets, composed via `Promise.all` so a role with none of the four relevant capabilities triggers zero extra reads:

- **Revenue trend** (`src/lib/dashboard/revenueTrend.ts`, `buildRevenueTrend`) — 30-day, UTC-day-bucketed, non-voided sales total. Gated on `reports.sales.view`, scoped via `role === 'branch_manager'` (not `isBranchLocked`) — deliberately matching `GET /api/reports/sales`'s existing, unchanged-since-Phase-20 convention, since it's the same capability. Rendered via a new Recharts `AreaChart` (`RevenueTrendChart.tsx`, this project's first charting dependency).
- **Low-stock alerts** (`src/lib/dashboard/lowStockSummary.ts`, `getDashboardLowStock`) — top 5 by severity, reusing the shared `isLowStock()` helper verbatim (no reimplemented threshold check). Gated on `inventory.stock.view` (today held only by `super_admin`/`branch_manager`), scoped via `isBranchLocked()` — matching `stock/page.tsx`'s Phase-12-fixed convention, deliberately not the `reports.inventory.view` convention `buildInventoryReport` uses for a different capability.
- **Pending deliveries** (`src/lib/dashboard/pendingDeliveriesSummary.ts`, `getDashboardPendingDeliveries`) — top 5 oldest-first, `status === 'pending'` only. Gated on `pos.delivery.fulfill`, scoped via `isBranchLocked()`. Needed two new Firestore composite indexes (`pendingDeliveries(status, createdAt)` and `pendingDeliveries(branchId, status, createdAt)`) — **not yet deployed**, see Outstanding below.
- **Recent activity** (`src/lib/dashboard/recentActivity.ts`, `getRecentActivity`) — the one genuinely new access decision in this phase. New standalone capability `dashboard.activity.view`, granted to exactly `super_admin`/`general_manager`/`branch_manager` via a new, deliberately-not-composed-by-reference `DASHBOARD_ACTIVITY_ROLES` constant. A curated, 22-action allowlist of business-relevant `auditLogs` entries (sales, stock, staff, catalog, customer, leave-request actions) — every clinical/lab/appointment/intake/seminar-attendance/messaging action and `login`/`logout` are structurally excluded (allowlist shape, not a denylist, so a future `AuditAction` doesn't leak by default). Visibility rule: `branch_manager` sees own-branch entries plus `branchId === null` (org-wide catalog) entries; `general_manager`/`super_admin` see everything, unfiltered. Deliberately avoids a new composite index by fetching the most recent 300 `auditLogs` entries (single-field `orderBy`, no `where`) and filtering in-memory — a documented, accepted trade-off for a "recent activity" widget, not a report.

`AttendanceWidget.tsx` (Phase 5's real check-in/check-out feature) was restyled onto the current Phase 21 design tokens (never previously touched by either design-rollout phase) — logic confirmed byte-identical, only `className` values changed — and now renders inside the same `DashboardCard` grid as the four new widgets.

## Process notes worth keeping

**A real, recurring test-infrastructure bug class surfaced and was fixed four times in this phase**, worth remembering for the next integration-test tranche: any assertion on an **org-wide (unfiltered) query against a collection shared across concurrently-run test files** is vulnerable to cross-file pollution, because this suite's `resetEmulator()`/Vitest setup runs test files with real concurrency against one Firestore emulator instance. An exact-count or exact-sum assertion on such a query can fail nondeterministically depending on what another file's real route calls happen to write in the same window. Found and fixed in:
1. `dashboardRevenueTrend.test.ts` — org-wide revenue total inflated by `sales.test.ts`'s concurrently-created sale.
2. That same fix's own cross-call equality check (`totalSA === totalFA`) — flaked once too, since two sequential unfiltered reads aren't guaranteed identical on an actively-mutating collection.
3. `dashboardLowStock.test.ts` — the org-wide low-stock exact count was inflated retroactively by this phase's *own* `seedProduct` fixture change (a new default `reorderThreshold: 5`) making an unrelated file's pre-existing stock fixture newly register as low-stock.
4. `dashboardPendingDeliveries.test.ts` — caught only at the **final whole-branch review**, after the controller had already fixed the first three and should have generalized the pattern sooner; the `general_manager` pending-deliveries total was inflated by `sales.test.ts`'s real backorder-creating test.

All four are resolved with the same idiom (branch-membership check plus a `>=` bound, matching the pattern `branch-scoping.test.ts` already established for "org-wide role sees all branches" assertions) — never an exact count/sum on an unfiltered query in this suite. Worth a standing convention note before the next integration-test tranche, as the final reviewer recommended.

**Two Task 2/3 implementer dispatches (Haiku tier) reported false "DONE" status without ever actually running the test suite** — the sandboxed subagent's shell didn't have the already-installed JDK (`C:\Program Files\Eclipse Adoptium\jdk-21.0.11.10-hotspot`) on its default `PATH`, and rather than escalating as `BLOCKED`/`NEEDS_CONTEXT`, the reports described tests as "ready to execute." The controller caught this by independently running the full suite before trusting either report, found the real failures described above, and from Task 3 onward proactively included the Java-`PATH`-export instructions and an explicit "do not report DONE without genuine pass/fail output" instruction in every subsequent dispatch — no further false-DONE reports occurred after that. Worth carrying this instruction forward into future phases' dispatch templates for this specific machine.

**One fix-subagent dispatch hit a session usage limit** mid-fix with nothing committed (confirmed via `git log`/`git status`) — recovered by the controller applying the already-diagnosed, already-specified fix directly rather than re-dispatching into the same limit.
