# Phase 18.1 — Offline Queue & Sync Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let checkout work with no network connectivity at all — a sale completed offline is queued locally and shown as a provisional receipt, then replayed through the existing `POST /api/sales` once connectivity returns, one at a time, with a server-side idempotency check preventing a double-sale if a sync response is ever lost after the server already committed.

**Architecture:** IndexedDB holds two things: a cached snapshot of the catalog (products/services, written whenever the device successfully loads them online) and the offline sale queue itself (one record per queued sale, keyed by a client-generated idempotency key that's created once and reused on every retry). `CheckoutForm` generates that key at submit time regardless of path, attempts a live `POST /api/sales` when it believes it's online, and falls back to enqueueing on any network-level failure (including a false-positive "online" signal). A sync engine replays `queued` entries sequentially in creation order on reconnect/periodic tick, distinguishing a network failure (retry later, don't touch the rest of the queue this pass) from a real server rejection (mark that one entry `needs_attention`, keep going). `api/sales/route.ts` gains one new check — an idempotency-key lookup, inside the same transaction, before any other read — as its only change; price resolution, stock-decrement, and backorder logic from Phase 18 are untouched.

**Tech Stack:** Same as the rest of the app — Next.js Client Components, browser `IndexedDB` (hand-written thin wrapper, no new npm dependency — this project currently has 5 runtime dependencies and this need doesn't justify a 6th), Firebase Admin SDK/Firestore for the one server-side change. No unit test suite in this project (confirmed convention) — verification per task is `npx tsc --noEmit` plus manual browser/`fetch`-based checks; live verification against real `erp-lfd` data closes the phase, gated on the user's go-ahead.

## Global Constraints

- Scope is one device, one browser tab — no multi-device conflict resolution, no background sync while the tab is closed, no service worker / PWA shell. A hard page reload with zero connectivity is out of scope (would need a service worker to intercept the navigation request at all); "usable offline" means within a continuously-open tab.
- Full-quantity-only sales (Phase 18's own scope) is unchanged — this phase doesn't touch partial fulfillment.
- `api/sales/route.ts`'s price resolution, stock-decrement, and backorder logic must be provably unchanged — the only new code is the idempotency-key check, and it must run before any other read inside the transaction.
- Queued sales must sync strictly in creation order, one at a time — never in parallel.
- The idempotency key is generated once per checkout attempt and reused on every retry of that same attempt (including the very first live attempt, not just queued retries) — this is what actually closes the "response lost after the server committed" gap.

## Decisions requiring your sign-off before implementation

### 1. The idempotency key is generated at submit time and sent on the FIRST live attempt too, not only on queued retries

If the key were only attached when a sale is already known to be queued, the scenario the brief calls out — a live attempt that actually succeeds server-side but whose response is lost — would go completely unprotected: the first (uncounted) attempt would create a sale with no key, and the later queued retry (now carrying a key) would find no match and create a second sale. Generating the key once, up front, and including it on every attempt (live or queued) for that same cart is the only way the dedupe check can ever catch this case. `CheckoutForm` always sends `clientIdempotencyKey` on every `POST /api/sales` call it makes, full stop.

**Resolved as above unless you object.**

### 2. Sync loop: a network-level failure stops that pass; a server rejection continues to the next item

These are different failure classes and the brief treats them differently. A `fetch` that *throws* (no response reached the client at all) means the network is genuinely still down — every other queued item would fail identically right now, so the sync pass stops there and retries from the same point on the next `online` event or periodic tick, in the same order. A `fetch` that *resolves* with a non-2xx status means the server was reached and gave a definitive answer (most commonly the backorder-without-customer 409) — that one entry becomes `needs_attention` and the loop continues to the next queued item, per the brief's explicit "does not block the rest of the queue behind it."

**Resolved as above unless you object.**

### 3. Hand-written IndexedDB wrapper, not a new dependency

This project has exactly 5 runtime dependencies (`firebase`, `firebase-admin`, `next`, `react`, `react-dom`) and no existing precedent for a client-storage helper library. The need here (two object stores, basic CRUD) doesn't justify adding `idb` or similar — a small hand-written promise wrapper over the raw `indexedDB` API covers it.

**Resolved as above unless you object.**

### 4. Catalog caching is write-through within the current tab session only — no service worker

"The catalog and stock levels are usable offline from the last successful cache" is satisfied by: `CheckoutForm` writes the catalog to IndexedDB whenever it mounts while online, and its in-memory React state (already loaded from server props) continues to serve search/select with zero network calls for the rest of that tab's life, online or offline — no code change is even needed for that part, since `CheckoutForm` never re-fetches products/services after its initial server-rendered load. The IndexedDB cache's job is narrower: recording *when* the catalog was last synced, so the UI can show that timestamp in an offline banner. A hard reload with no connectivity at all is explicitly out of scope per the Global Constraints — that needs a service worker, a different and larger undertaking not asked for here.

**Resolved as above unless you object.**

### 5. Queue visibility is gated to `cashier`/`branch_manager` in the nav shell

The brief says "surface it clearly to `cashier`/`branch_manager`" — these are the only two roles that create sales at all (`CASHIER_BRANCH_MGR` backs `pos.sale.create`). The queue indicator renders in `NavShell.tsx` only for those two roles, not for every signed-in user.

**Resolved as above unless you object.**

### 6. Resolving a `needs_attention` sale fetches the customer list from the existing `GET /api/customers` route, on demand

Attaching a customer to a `needs_attention` sale and retrying it inherently requires connectivity (you're about to make a network request either way) — there's no offline-first requirement for the resolution UI itself. It lazily fetches `GET /api/customers` (already capability-gated, already used by `CheckoutForm`'s own customer picker) the first time a user opens the "attach customer" control for any entry, rather than requiring a separate offline-customer cache.

**Resolved as above unless you object.**

### 7. Synced entries are auto-cleared after a short grace period, not by a manual dismiss button

"Keep the record briefly for the UI to show recent sync activity, then it can be cleared" is implemented as: any time the queue is polled for display, entries with `status: 'synced'` older than 60 seconds are deleted as a light housekeeping step. No separate "dismiss" button needed.

**Resolved as above unless you object.**

```ts
// Thin promise wrapper over the raw IndexedDB API — this project has no
// existing client-storage dependency and the need here (two object stores,
// basic CRUD) doesn't justify adding one. Every function in this module
// must only ever be called from client-side code (event handlers, effects)
// — IndexedDB doesn't exist during Next.js server rendering.

const DB_NAME = 'lfd-pos-offline'
const DB_VERSION = 1
export const QUEUE_STORE = 'queuedSales'
export const CATALOG_STORE = 'catalogCache'

let dbPromise: Promise<IDBDatabase> | null = null

export function openOfflineDb(): Promise<IDBDatabase> {
  if (dbPromise) return dbPromise
  dbPromise = new Promise((resolve, reject) => {
    const req = indexedDB.open(DB_NAME, DB_VERSION)
    req.onupgradeneeded = () => {
      const db = req.result
      if (!db.objectStoreNames.contains(QUEUE_STORE)) {
        db.createObjectStore(QUEUE_STORE, { keyPath: 'idempotencyKey' })
      }
      if (!db.objectStoreNames.contains(CATALOG_STORE)) {
        db.createObjectStore(CATALOG_STORE, { keyPath: 'branchId' })
      }
    }
    req.onsuccess = () => resolve(req.result)
    req.onerror = () => reject(req.error)
  })
  return dbPromise
}

export async function dbPut<T>(storeName: string, value: T): Promise<void> {
  const db = await openOfflineDb()
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readwrite')
    tx.objectStore(storeName).put(value)
    tx.oncomplete = () => resolve()
    tx.onerror = () => reject(tx.error)
  })
}

export async function dbGet<T>(storeName: string, key: IDBValidKey): Promise<T | undefined> {
  const db = await openOfflineDb()
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readonly')
    const req = tx.objectStore(storeName).get(key)
    req.onsuccess = () => resolve(req.result as T | undefined)
    req.onerror = () => reject(req.error)
  })
}

export async function dbGetAll<T>(storeName: string): Promise<T[]> {
  const db = await openOfflineDb()
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readonly')
    const req = tx.objectStore(storeName).getAll()
    req.onsuccess = () => resolve(req.result as T[])
    req.onerror = () => reject(req.error)
  })
}

export async function dbDelete(storeName: string, key: IDBValidKey): Promise<void> {
  const db = await openOfflineDb()
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readwrite')
    tx.objectStore(storeName).delete(key)
    tx.oncomplete = () => resolve()
    tx.onerror = () => reject(tx.error)
  })
}
```

```ts
  await dbDelete(QUEUE_STORE, idempotencyKey)

// recent activity, then cleared automatically the next time the queue is
// listed for display — see Decision #7.
const SYNCED_RETENTION_MS = 60_000

export async function pruneSyncedEntries(): Promise<void> {
  const all = await listQueuedSales()
  const now = Date.now()
  for (const item of all) {
    if (item.status === 'synced' && now - item.createdAt > SYNCED_RETENTION_MS) {
      await deleteQueuedSale(item.idempotencyKey)
    }
  }
```

```ts
'use client'
import { useEffect, useState } from 'react'

// Starts true to avoid a hydration mismatch (the server has no concept of
// "online" — it always renders as if connected); useEffect corrects it
// immediately on mount, before the user can interact with anything.
export function useOnlineStatus(): boolean {
  const [isOnline, setIsOnline] = useState(true)

  useEffect(() => {
    setIsOnline(navigator.onLine)
    const goOnline = () => setIsOnline(true)
    const goOffline = () => setIsOnline(false)
    window.addEventListener('online', goOnline)
    window.addEventListener('offline', goOffline)
    return () => {
      window.removeEventListener('online', goOnline)
      window.removeEventListener('offline', goOffline)
    }
  }, [])

  return isOnline
}
```

Create `src/lib/pos/syncQueue.ts`:

```ts
import { listQueuedSales, updateQueuedSale, pruneSyncedEntries, type QueuedSale } from './offlineQueue'

let syncing = false

export function isSyncing(): boolean {
  return syncing
}

// Replays queued sales through POST /api/sales, one at a time, in the
// order they were created. Two distinct failure classes, per this phase's
// Decision #2:
//   - fetch() itself throws (no response reached the client at all) — the
//     network is genuinely still down. Every remaining item would fail
//     identically right now, so this pass stops here; the item is reset to
//     'queued' and the next 'online' event or periodic tick retries from
//     the same point, in the same order.
//   - fetch() resolves with a non-2xx status — the server was reached and
//     gave a definitive answer (most commonly the backorder-without-
//     customer 409). That one entry becomes 'needs_attention' and the loop
//     continues to the next queued item — a rejection must not block the
//     rest of the queue.
export async function runSync(): Promise<void> {
  if (syncing) return
  syncing = true
  try {
    await pruneSyncedEntries()
    const all = await listQueuedSales()
    const toSync = all.filter((s: QueuedSale) => s.status === 'queued').sort((a, b) => a.createdAt - b.createdAt)

    for (const item of toSync) {
      await updateQueuedSale(item.idempotencyKey, { status: 'syncing' })

      let res: Response
      try {
        res = await fetch('/api/sales', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ ...item.payload, clientIdempotencyKey: item.idempotencyKey }),
        })
      } catch {
        await updateQueuedSale(item.idempotencyKey, { status: 'queued' })
        break
      }

      if (res.ok) {
        const body = await res.json()
        await updateQueuedSale(item.idempotencyKey, { status: 'synced', serverSaleId: body.id })
      } else {
        const body = await res.json().catch(() => ({}))
        await updateQueuedSale(item.idempotencyKey, {
          status: 'needs_attention',
          lastError: body.error ?? `Server rejected the sale (HTTP ${res.status})`,
        })
      }
    }
  } finally {
    syncing = false
  }
}

// Called after a user attaches a customer to a needs_attention sale — puts
// it back at the front of the normal queue (status 'queued') with the
// customerId now set, then immediately attempts a sync.
export async function resolveNeedsAttention(idempotencyKey: string, customerId: string): Promise<void> {
  const all = await listQueuedSales()
  const item = all.find((s) => s.idempotencyKey === idempotencyKey)
  if (!item) return
  await updateQueuedSale(idempotencyKey, {
    payload: { ...item.payload, customerId },
    status: 'queued',
    lastError: null,
  })
  await runSync()
}
```

```tsx
  name: string
  phone: string
```
