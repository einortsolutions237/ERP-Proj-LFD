# Phase 4 тАФ CRM Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Customer records (org-wide, same pattern as products/services/suppliers) with phone-uniqueness, an optional link from a sale to a customer, and a purchase-history view тАФ with the smallest possible touch to the already-twice-reviewed sale-creation transaction.

**Architecture:** Same modular monolith as Phases 1тАУ3.1. One new collection (`customers`), one new optional field on `sales` (`customerId`), three new capabilities. No change to `stockMovements`, `productStock`, or the void mechanic тАФ this phase doesn't touch inventory.

**Tech Stack:** Unchanged тАФ Next.js App Router, TypeScript, Tailwind, Firebase Auth + Firestore + Admin SDK.

## Global Constraints

- Firestore rules default-deny; every collection needs an explicit rule
- Every write goes through an Admin-SDK-backed API route with an explicit field whitelist тАФ never `{...body}` spread
- Firebase custom claims are the sole source of truth for authorization
- Phone uniqueness on `customers` is enforced inside a real Firestore transaction (read-check-write atomic) тАФ this is exactly the race the Phase 2 audit flagged on product SKUs; do not repeat a plain-read-then-separate-write here
- Every customer create/edit/delete produces an audit log entry
- No TDD / no automated test suite тАФ manual verification against exit criteria, matching Phases 1тАУ3.1
- **`src/app/api/sales/route.ts` is already-shipped, twice-reviewed code (Phase 3's original transaction, Phase 3.1's void-fields amendment).** This phase's ONLY change to it is accepting and validating one new optional field (`customerId`). No other line changes тАФ no reordering, no refactor, no "while I'm in here" cleanup. Treat price resolution, stock decrement, and payment-sum validation as already correct and out of scope to touch.

## `customers` collection

**`customers/{customerId}`** тАФ org-wide, auto-ID (same non-branch-scoped treatment as `products`/`services`/`suppliers` тАФ see CLAUDE.md's "Org-wide entities" section, added as part of this planning pass):
```ts
{
  name: string
  phone: string              // the unique identifier
  email: string | null
  address: string | null
  notes: string | null
  registeredBranchId: string  // which branch first added them тАФ reference only, never used for access control or filtering, same role supplierId-style fields play elsewhere
  createdAt: Timestamp
  updatedAt: Timestamp        // matches every other org-wide entity's shape (products/suppliers both have this) тАФ not explicit in your scope text, but a direct, consistent inference from "editable" implying a last-modified timestamp
}
```

**Phone uniqueness тАФ the exact pattern already fixed once in this codebase (Phase 3's carried-over Fix A1 on product SKUs), applied from the start here rather than needing a later fix:**
```ts
const newCustomerRef = db.collection('customers').doc()
await db.runTransaction(async (tx) => {
  const phoneSnap = await tx.get(db.collection('customers').where('phone', '==', phone).limit(1))
  if (!phoneSnap.empty) {
    throw new AuthError('A customer with this phone number already exists', 409)
  }
  tx.set(newCustomerRef, { name, phone, email, address, notes, registeredBranchId: user.branchId, createdAt: new Date(), updatedAt: new Date() })
})
```
On edit (`PATCH`), only re-run the uniqueness query if `phone` is actually being changed, and exclude the document being edited from the "would collide" check тАФ the exact self-exclusion logic already reviewed and approved for product SKU edits in Phase 2.

**Referential delete guard (judgment call, flagging per this codebase's established convention rather than silently deciding):** following the same pattern as branchesтЖТstaff/departments and suppliersтЖТproducts, block deleting a customer if any `sales` doc has `customerId == id` (409 `'Cannot delete a customer that is still referenced by a sale'`). Not explicit in your scope text, but a direct, consistent application of a pattern used everywhere else a delete could orphan a reference.

## `sales.customerId` тАФ the sale-creation amendment

Add `customerId: string | null` to the `Sale` type (`src/lib/types/sale.ts`) тАФ always present (like `voidedAt`/`voidedBy`/`voidReason`), never omitted.

**The amendment to `src/app/api/sales/route.ts`, precisely scoped тАФ four small additions, nothing else touched:**

1. **After the existing `payments` validation block, before `const db = getAdminFirestore()`:** validate the optional `customerId` from the request body.
```ts
let customerId: string | null = null
if ('customerId' in body && body.customerId !== undefined && body.customerId !== null) {
  if (!isNonEmptyString(body.customerId)) {
    return NextResponse.json({ error: 'customerId must be a non-empty string or null' }, { status: 400 })
  }
  customerId = body.customerId.trim()
}
```

2. **Inside the transaction, in the READS section** (grouped alongside the existing item-resolution reads, before `resolvedLineItems` is computed): resolve and validate the customer, if one was supplied.
```ts
let customerSnap: FirebaseFirestore.DocumentSnapshot | null = null
if (customerId) {
  customerSnap = await tx.get(db.collection('customers').doc(customerId))
  if (!customerSnap.exists) {
    throw new AuthError('customerId does not reference an existing customer', 400)
  }
}
```
This is one more read before any write, same rule the rest of the transaction already follows тАФ no reordering of the existing reads is needed, this is additive.

3. **In the existing `tx.set(saleRef, {...})` write:** add `customerId,` to the object literal (alongside `voidedAt: null, voidedBy: null, voidReason: null,` тАФ same "fully populate the shape" convention).

4. **In the `writeAuditLog(...)` call's `details`:** add `customerId,` alongside the existing `lineItems`/`subtotal`/`total`/`payments` fields тАФ a sale's audit entry already carries full detail; the customer link is part of that detail now.

Nothing else in this file changes. `resolvedLineItems`, `subtotal`/`discountAmount`/`taxAmount`/`total` computation, `paymentsSum` validation, the stock-aggregation/sufficiency-check loop, and the stock-write loop are all untouched.

**Deliberately NOT in scope for this phase (flagging rather than silently deciding either way):** the receipt page (`/pos/sales/[id]`) and sales list (`/pos/sales`) are not modified to display the linked customer's name. Your scope text specifies the *customer's* detail view shows purchase history (one direction); it doesn't ask for the *sale's* view to show the customer (the other direction). Both are cheap to add later if wanted тАФ leaving them out now keeps this phase's UI surface exactly as scoped.

## Checkout integration тАФ CheckoutForm

**Design (matching an existing pattern, not inventing a new one):** the existing checkout page already fetches the full `products`/`services` lists in its server component and filters them client-side in `CheckoutForm` тАФ no dedicated search API exists for those, and customers get the identical treatment for consistency, not a new debounced-search-API pattern. `src/app/(dashboard)/pos/page.tsx` additionally fetches `customers` (a lightweight projection: `id`, `name`, `phone` тАФ full record isn't needed for the picker) and passes it to `CheckoutForm`.

**In `CheckoutForm.tsx`, add a "Customer" section** (a new `customerId: string | null` state field, `customerSearch: string` state for the picker's own search box, separate from the existing product/service search):
- Default state: no customer selected тАФ "Walk-in" is the visible default, not an empty-looking gap. The sale submits with no `customerId` in the request body in this state, identical to today's behavior (Phase 3 regression-safety requirement).
- A search box filtering the passed-in customer list by name or phone (client-side `.filter()`, same mechanism as the existing product/service search). Selecting a result sets `customerId`.
- A "Quick add" affordance: two inputs (name, phone тАФ nothing else, per your scope: "the full field set lives on the dedicated customer screens, not squeezed into checkout"). Submitting it calls `POST /api/customers` with just `{name, phone}` (email/address/notes omitted, stored `null` server-side per the existing optional-field convention). On success, the newly created customer is immediately selected for the current cart (`customerId` set to the response's new id) тАФ no extra round trip or page navigation.
- A visible "Remove" / "Change customer" affordance once one is selected, returning to the walk-in default.
- On submit, include `customerId` in the `POST /api/sales` body only when one is selected (omit the key entirely when walk-in, matching the route's `'customerId' in body` check).

**Capability for the quick-add call:** `crm.customer.create` тАФ since `cashier` needs to reach this at checkout, and `crm.customer.create` is the capability your scope grants to `cashier`.

## Permissions

Add to `src/lib/auth/permissions.ts`'s `Capability` union and `ROLE_CAPABILITIES`, under the already-reserved `crm` module (remove `crm.*` from the placeholder comment, exactly as `inventory.*` and `pos.*` were removed from it in earlier phases):
```ts
'crm.customer.create'   // super_admin, admin, branch_manager, cashier
'crm.customer.view'     // super_admin, admin, branch_manager, cashier
'crm.customer.manage'   // edit/delete тАФ super_admin, admin, branch_manager ONLY, not cashier
```
`crm.customer.create`/`crm.customer.view` reuse the existing `CASHIER_BRANCH_MGR` constant (identical role set to `pos.sale.create`/`pos.sale.view`). `crm.customer.manage` reuses the existing `ADMIN_BRANCH_MGR` constant (identical role set to `pos.sale.void`). No new role-group constants needed тАФ both already exist.

**Note this is a genuinely different shape than products/suppliers, which bundle create+edit+delete under one capability (`inventory.catalog.manage`, `inventory.suppliers.manage`) тАФ not an inconsistency to "fix," a deliberate three-way split because `cashier` needs to create customers at checkout but must never edit or delete one, a trust boundary those other entities never needed since `cashier` never touches them at all.**

## Audit Actions

Add `'customer_create' | 'customer_edit' | 'customer_delete'` to `AuditAction` in `src/lib/types/audit.ts`, following the exact naming pattern of `supplier_create`/`supplier_edit`/`supplier_delete`. `branchId: null` on all three (org-wide entity, same convention as every other org-wide entity's audit entries) тАФ `registeredBranchId` is reference metadata on the document itself, not what gates the audit log's own `branchId` field.

## API Routes

**`src/app/api/customers/route.ts`**
- `GET` тАФ `requireCapability('crm.customer.view')`, unfiltered (org-wide, matching `products`/`suppliers`'s exact GET pattern тАФ no `branchId` filter).
- `POST` тАФ `requireCapability('crm.customer.create')`. Validates `name` (required non-empty string), `phone` (required non-empty string, uniqueness enforced transactionally as above), `email`/`address`/`notes` (each optional string-or-null, same validation shape already reviewed for suppliers' contact fields). `registeredBranchId: user.branchId` тАФ server-derived, never client-supplied. Audit `customer_create`, `branchId: null`.

**`src/app/api/customers/[id]/route.ts`**
- `PATCH` тАФ `requireCapability('crm.customer.manage')`. Partial update; phone uniqueness re-checked only if `phone` is present in the body and differs from the existing value, self-excluded (same pattern as products' SKU edit). Audit `customer_edit`, `branchId: null`.
- `DELETE` тАФ `requireCapability('crm.customer.manage')`. Referential guard against `sales.customerId` as above (409 if referenced). Audit `customer_delete`, `branchId: null`.

## Screens

**`src/app/(dashboard)/customers/page.tsx`** тАФ list/search (server component): `requireCapability('crm.customer.view')`, fetch all customers, render a searchable table (name, phone, email, a link to detail) тАФ client-side filter by name/phone, same mechanism as every other list-plus-search screen in this app. An "Add customer" link to `/customers/new`, visible only if `hasCapability(user.role, 'crm.customer.create')`.

**`src/app/(dashboard)/customers/new/page.tsx`** тАФ create form: `requireCapability('crm.customer.create')` (not `manage` тАФ `cashier` can reach this screen too, per your scope's "via checkout quick-add or the full screen"). Full field set: name, phone, email, address, notes.

**`src/app/(dashboard)/customers/[id]/page.tsx`** тАФ read-only detail + purchase history: `requireCapability('crm.customer.view')`. Fetches the customer doc (404 if missing) and their purchase history: `sales` where `customerId == id` **AND `branchId == user.branchId`** тАФ the same branch-scoping every other branch-scoped list in this app already applies (there is no "admin sees all branches" special case anywhere in this codebase; a customer is org-wide, but the *sales* referencing them are branch-scoped data, and this view follows the existing convention of scoping branch-scoped reads to the viewer's own branch, not a new cross-branch exception). Renders customer info read-only, plus a purchase history table matching the sales list's level of detail (date/time, item count, total, payment methods, link to the sale's receipt). Shows "Edit" and "Delete" controls only if `hasCapability(user.role, 'crm.customer.manage')` тАФ mirrors the exact capability-gated-control pattern already reviewed for `VoidSaleButton` on the sale receipt page in Phase 3.1. `cashier` (who has `view` but not `manage`) sees the same read-only page with no edit/delete controls, not a different/degraded page.

**`src/app/(dashboard)/customers/[id]/edit/page.tsx`** тАФ the edit form: `requireCapability('crm.customer.manage')`. A separate page rather than folding edit-mode into the detail page тАФ this app has no existing precedent for "one page, two render modes gated by different capabilities" beyond the small `VoidSaleButton` toggle; a full CRUD form is enough larger/different in shape that a dedicated page (matching the `departments`/`suppliers`/`products` convention of a plain `[id]/page.tsx`-is-the-edit-form pattern) is more consistent with this codebase's existing structure than cramming a conditional form into the read-only detail page.
