# Phase 3 — POS Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ring up a sale of products and/or services, take a manually-recorded payment (cash / MTN Mobile Money / Orange Money, any split), and correctly move stock — reusing Phase 2's stock-transaction mechanic exactly, not a new one.

**Architecture:** Same modular monolith as Phases 1–2. One new collection (`sales`), one new movement type (`sale`) on the existing `stockMovements` ledger, one new role (`cashier`) on the existing client-SDK login path, two new capabilities (plus one reserved-but-unimplemented). No new subsystems.

**Tech Stack:** Unchanged — Next.js App Router, TypeScript, Tailwind, Firebase Auth + Firestore + Admin SDK.

## Global Constraints

- Firestore rules default-deny; every collection needs an explicit rule — no blanket permissive rules
- Every write goes through an Admin-SDK-backed API route with an explicit field whitelist — never `{...body}` spread
- Firebase custom claims are the sole source of truth for authorization; never read Firestore to make an access-control decision
- Enforce role + branch checks at UI, server, and Firestore rules — all three, for every new capability
- No `Math.random()` for security-sensitive values — use a CSPRNG
- No hardcoded credentials or `??` fallback literals in client code
- Every completed sale produces an audit log entry via the existing `writeAuditLog()`, with full details (not the bare pattern Phase 2 used for catalog edits)
- No TDD / no automated test suite — manual verification against exit criteria, matching Phases 1–2
- No payment-gateway or mobile-money API integration of any kind, anywhere — payment method is a manually-entered record, never verified against MTN/Orange/any processor
- Void/refund is explicitly out of scope this phase — no code path may undo, edit, or reverse a completed sale

### Fix A1: Wrap product SKU-uniqueness check in a real transaction

**File:** `src/app/api/products/route.ts`

**Problem:** the current `POST` handler does a plain read (`where('sku','==',sku).limit(1).get()`) followed by a separate `add()` — two independent round-trips with a race window between them where two concurrent creates of the same SKU could both pass the uniqueness check.

**Fix:** wrap the check-and-write in `db.runTransaction()`, using a transactional query read (`tx.get(query)`) exactly the way `src/app/api/stock/movements/route.ts` and `src/app/api/stock/transfer/route.ts` already use `tx.get()` for transactional reads before any write. Firestore retries the transaction if a matching document is created concurrently between the read and commit, closing the race.

- [ ] **Step 1: Replace the create handler's SKU-check/write with a transaction**

Replace this block in `POST` (currently):
```ts
const db = getAdminFirestore()
const sku = body.sku.trim()

const skuSnap = await db.collection('products').where('sku', '==', sku).limit(1).get()
if (!skuSnap.empty) {
  return NextResponse.json({ error: 'A product with this SKU already exists' }, { status: 409 })
}

const productData = {
  name: body.name.trim(),
  sku,
  category: body.category.trim(),
  unitCost: body.unitCost,
  price: body.price,
  supplierId: isNonEmptyString(body.supplierId) ? body.supplierId : null,
  reorderThreshold: body.reorderThreshold,
  active: true,
  createdAt: new Date(),
  updatedAt: new Date(),
}
const docRef = await db.collection('products').add(productData)

await writeAuditLog({ action: 'product_create', actorUid: user.uid, actorEmail: user.email, targetUid: docRef.id, branchId: null })

return NextResponse.json({ id: docRef.id }, { status: 201 })
```

With:
```ts
const db = getAdminFirestore()
const sku = body.sku.trim()
const newProductRef = db.collection('products').doc()

try {
  await db.runTransaction(async (tx) => {
    const skuSnap = await tx.get(db.collection('products').where('sku', '==', sku).limit(1))
    if (!skuSnap.empty) {
      throw new AuthError('A product with this SKU already exists', 409)
    }
    tx.set(newProductRef, {
      name: body.name.trim(),
      sku,
      category: body.category.trim(),
      unitCost: body.unitCost,
      price: body.price,
      supplierId: isNonEmptyString(body.supplierId) ? body.supplierId : null,
      reorderThreshold: body.reorderThreshold,
      active: true,
      createdAt: new Date(),
      updatedAt: new Date(),
    })
  })
} catch (err) {
  if (err instanceof AuthError) return NextResponse.json({ error: err.message }, { status: err.status })
  throw err
}

await writeAuditLog({ action: 'product_create', actorUid: user.uid, actorEmail: user.email, targetUid: newProductRef.id, branchId: null })

return NextResponse.json({ id: newProductRef.id }, { status: 201 })
```

Note the outer `try/catch` at the top of `POST` already catches `AuthError` from `requireCapability` — this inner `try/catch` is a second, separate one scoped just around the transaction, matching exactly how `stock/movements/route.ts` and `stock/transfer/route.ts` structure theirs (an inner try/catch around `runTransaction` that re-throws anything that isn't an `AuthError`, inside the outer route-level try/catch). Leave the rest of the validation block (name/sku/category/unitCost/price/reorderThreshold/supplierId checks) above this untouched.

## Part B: POS

## Firestore Collections

**`sales/{saleId}`** — branch-scoped, auto-ID
```ts
{
  branchId: string
  lineItems: Array<{
    type: 'product' | 'service'
    itemId: string        // productId or serviceId
    name: string           // snapshot at sale time — never a live catalog reference
    unitPrice: number      // snapshot at sale time — never a live catalog reference
    quantity: number
    lineTotal: number      // unitPrice * quantity
  }>
  subtotal: number         // sum of lineTotals
  discountAmount: number   // flat amount, >= 0, defaults 0
  taxAmount: number        // stub — always literal 0 this phase, never client-supplied
  total: number             // subtotal - discountAmount + taxAmount
  payments: Array<{
    method: 'cash' | 'mtn_momo' | 'orange_money'
    amount: number
    reference: string | null   // free text the cashier read off their own phone; never validated against anything
  }>
  cashierUid: string
  createdAt: Timestamp
}
```

**Snapshot pricing, not client-supplied pricing (judgment call, flagging rather than silently deciding):** the client only ever sends `{type, itemId, quantity}` per line — never `name`/`unitPrice`. The server looks up the current `products`/`services` doc for each `itemId` and takes `name`/`price` from there at the moment of sale; that resolved value becomes the stored snapshot. This is the only way to satisfy "never trust client input for a financial write" (an existing Global Constraint) while also satisfying the spec's "not a live reference to the catalog" requirement (the *sale* doc doesn't re-read the catalog after creation — it's a snapshot from the moment of sale, just resolved server-side rather than client-side).

**Item must be active:** a line item whose `products`/`services` doc has `active: false` is rejected (400) — you cannot sell a discontinued/inactive catalog entry. Not explicit in the spec text, but a direct consequence of what `active` already means everywhere else in this codebase.

**`stockMovements`** — extend the existing type, don't create a new collection:
```ts
export type StockMovementType = 'restock' | 'adjustment' | 'waste' | 'transfer_out' | 'transfer_in' | 'sale'
```
Add one new field to the `StockMovement` interface, parallel to the existing `transferId`:
```ts
saleId: string | null   // set (to the sales/{saleId} doc id) only for type:'sale'; null for every other type
```
`quantityDelta` for a `sale` movement is negative (stock leaving), same sign convention as `waste`. `reason` is `null` for sale movements — the `saleId` link is the traceable reference, not free text. Existing Phase 2 movement docs predate this field and simply don't have it; nothing reads `saleId` on them, so this is a no-op schema addition, not a migration.

## Stock Decrement Mechanic — how a sale hooks into the existing transaction pattern

This reuses `stock/movements/route.ts`'s exact pattern (read current quantity inside the transaction → reject if the result would go negative → atomic `FieldValue.increment` → movement doc created in the same transaction), extended in two ways a single-product movement didn't need to handle:

1. **The whole sale is one transaction, not one transaction per product.** The sale doc itself, every affected `productStock` doc, and every new `stockMovements` doc are all written inside a single `db.runTransaction()`. If any product line's stock would go negative, the transaction throws before anything commits — the sale doc is never created, no stock moves, no movement docs are written. This is what satisfies "reject the whole sale rather than partially completing it."

2. **Multiple cart lines can reference the same product — they must be aggregated before checking sufficiency, not checked line-by-line.** If a cart has product X on two separate lines (e.g. added at different points in the checkout flow) totaling quantity 6, and current stock is 5, checking each line independently against the same stale read (5 - 3 = 2, "ok"; 5 - 3 = 2 again, "ok") would incorrectly allow overselling by 1. The correct approach: **group line items by `itemId` in memory first** (plain JS, before touching Firestore), sum `quantity` per distinct product `itemId`, then for each distinct product do exactly one `tx.get(productStockRef)`, one sufficiency check against the *summed* quantity, one `FieldValue.increment(-summedQuantity)` write, and **one `stockMovements` doc per distinct product** (not one per cart line) with `quantityDelta` equal to the negative summed quantity. Service line items never enter this grouping at all — they're filtered out before the grouping step and never touch `productStock`/`stockMovements`.

Before the transaction starts: generate `const saleRef = db.collection('sales').doc()` (auto-ID, pre-generated the same way `movementRef`/`outMovementRef`/`inMovementRef` are pre-generated before `runTransaction()` in the existing stock routes) and, for each distinct product identified during grouping, a `stockMovements` doc ref (`db.collection('stockMovements').doc()`) — refs are cheap to create and don't require a read, so generating them up front (outside the transaction) is what makes them available for `tx.set()` calls inside it.

Transaction shape, in order (all reads before any writes, per Firestore's transaction rule):
```
1. Resolve every line item's current product/service doc (reads — done via tx.get on each products/services doc referenced by a line item, to get the authoritative name/price/active values)
2. Reject (throw AuthError 400) if any resolved item is missing or active:false
3. Group product-type line items by itemId, sum quantity per distinct productId
4. For each distinct productId: tx.get(productStock doc) — one read per distinct product
5. Compute resultingQuantity per distinct product; if any < 0, throw AuthError 409 "Insufficient stock for <product name>" — abort, nothing written
6. tx.set(saleRef, ...) with the fully resolved lineItems/subtotal/discountAmount/taxAmount/total/payments/cashierUid/branchId/createdAt
7. For each distinct product: tx.set(productStock, quantity: FieldValue.increment(-summedQuantity), merge:true) and tx.set(that product's pre-generated stockMovements ref, {..., type:'sale', quantityDelta:-summedQuantity, saleId: saleRef.id})
```
Steps 1 (the products/services reads) and 4 (the productStock reads) are both "reads" and must both happen before step 6/7's writes — straightforward since they're naturally sequenced that way already.

**Payment-sum validation:** `sum(payments[].amount)` must equal `total` — compare with a small epsilon tolerance (`Math.abs(paymentsSum - total) < 0.01`), not exact floating-point equality, since these are plain `number` currency values (this codebase has no cents-integer or decimal-library convention anywhere yet — matching existing `price`/`unitCost` handling, not introducing a new one). Reject with 400 if they don't match, before starting the transaction (no need to touch Firestore at all for a request that's malformed).

**`discountAmount` must not exceed `subtotal`** (judgment call — not explicit in the spec, but a total below zero makes no domain sense): reject with 400 if `discountAmount > subtotal`.

**`branchId` is never client-supplied** — always `user.branchId`, the acting cashier's own branch. Unlike stock-adjust (where `admin`/`super_admin` can act on any branch), a sale is physically tied to wherever the person ringing it up is — there's no "remote sale" concept. Simpler than the stock-adjust pattern, deliberately.

## Transaction Failure Handling

Every way `POST /api/sales` can fail, and what happens on each — this is a defined, closed set, not an open-ended "handle errors" instruction:

| Failure | Status | Response body | Retry behavior |
|---|---|---|---|
| Missing/malformed field (bad `lineItems` shape, non-numeric `amount`, unknown `method`, etc.) | 400 | `{error: '<specific field message>'}` | No server action taken; nothing was written. |
| `discountAmount > subtotal` | 400 | `{error: 'discountAmount cannot exceed subtotal'}` | Same — nothing written. |
| `sum(payments[].amount)` doesn't match `total` (outside epsilon) | 400 | `{error: 'Payments must sum to the sale total'}` | Same — nothing written. |
| A line item's `itemId` doesn't resolve to an existing, `active:true` product/service | 400 | `{error: 'Item <id> is not available for sale'}` | Same — nothing written. |
| Insufficient stock for one or more products (post-aggregation) | 409 | `{error: 'Insufficient stock for <product name>'}` | Transaction never commits — no sale doc, no stock movement, no `productStock` change for *any* line, not just the short one. |
| Not authenticated / lacks `pos.sale.create` | 401/403 | `{error: 'Not signed in'}` / `{error: 'Forbidden'}` | N/A — request rejected before touching Firestore. |
| Unexpected error (Firestore unreachable, etc.) | 500 | generic error | Nothing written (Firestore transactions don't partially commit). |

**No automatic retry anywhere in this stack, at any layer:**
- `db.runTransaction()`'s own internal retry (Firestore re-running the transaction callback if a document it read was concurrently modified) is a normal, invisible **infrastructure** mechanism scoped to a single in-flight request — it either eventually succeeds or eventually throws one of the errors above. This is not what "silent retry" refers to and isn't something the plan disables (it's how every other transaction in this codebase already behaves).
- What the plan explicitly rules out is **application-level** retry: `CheckoutForm` never automatically re-submits a failed sale. Every failure response above is surfaced to the cashier as a visible, specific error message; the cart state is preserved unmodified (nothing is cleared or reset on failure); the cashier must explicitly act — adjust the cart/payment and press "Complete Sale" again — to trigger a new, independent request. There is no code path where a failure is caught, swallowed, and silently retried without the cashier seeing it.

## Open questions / judgment calls (flagging per project convention, not guessing)

- **Snapshot pricing resolved server-side, not client-supplied** — flagged above; this is the only way to keep pricing tamper-proof given the existing "never trust client input for a write" constraint, but it does mean the checkout UI's displayed price and the eventually-stored price could theoretically differ if someone edits a product's price in another tab mid-checkout. Accepted as correct-by-design (the snapshot should reflect price *at the moment of sale*, which is exactly what a fresh server-side lookup gives you) rather than a bug.
- **Same-product-multiple-lines aggregation** — flagged in detail above; this is a real correctness pitfall for whoever implements Task 4, not optional guidance.
- **`branchId` always `user.branchId`, never client-supplied for a sale** — a deliberate simplification versus the stock-adjust/transfer pattern's per-request `branchId`, since a sale has no cross-branch concept the way a transfer does.
