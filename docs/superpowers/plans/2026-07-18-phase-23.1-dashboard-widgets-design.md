# Phase 23.1 — Dashboard Widgets (Role-Specialized) — Design

**Status:** approved, not yet implemented.

## Goal

Add three more widgets to the dashboard Phase 23 built (`src/app/(dashboard)/dashboard/page.tsx`): upcoming appointments, pending lab orders, pending leave approvals. Each surfaces data that already has a real query somewhere in this codebase — this phase is about composing/reusing existing logic on the dashboard, not building new aggregation.

Prerequisite confirmed: Phase 23 has landed and been code-reviewed clean (tagged `phase-23-baseline`, HEAD `2178efb` at the time this phase starts, working tree clean, nothing changed since).

Explicitly out of scope: additional widgets beyond these three; changes to the underlying schedule/worklist/approval queries themselves (`getAppointments`, `getPendingLabOrders`, `leave/review/page.tsx`'s inline query) beyond what's needed to make the leave-approval logic reusable (see section D).

## A. Page composition

`src/app/(dashboard)/dashboard/page.tsx` gets three more entries in its existing `Promise.all`, three more `hasCapability` booleans, three more conditional `DashboardCard` blocks — same shape as the four widgets already there. The four existing widgets' code is untouched.

## B. Upcoming appointments

Calls `getAppointments({ upcomingOnly: true }, user)` — the exact shared function `src/app/(dashboard)/appointments/page.tsx` and the customer-detail page already use. No new `doctorUid` filter: the existing schedule page defaults to org-wide (all doctors) unless a query param narrows it, so the widget matches that default rather than inventing self-filtering for a `doctor` viewer.

Gated on `clinical.appointments.manage` (`CLINICAL_VIEW_ROLES` = `super_admin`/`doctor`/`medical_secretary`/`general_manager` — confirmed `nurse` absent, both in the constant and by explicit live-verification in this phase's exit criteria). The query itself returns every future `status: 'scheduled'` appointment unbounded, already ascending-sorted by `scheduledAt`; the widget component slices to the soonest 5 — no change to `getAppointments`.

**Deliberate consequence, stated explicitly, not glossed over:** `getAppointments` audit-logs every call (`appointment_view`) by design — this project's clinical module has held "every clinical read is audited" as a load-bearing rule since Phase 13, specifically so broadening who holds a capability never broadens it without a trail. Reusing this function as-is means every dashboard load by a role holding `clinical.appointments.manage` now writes an `appointment_view` audit entry, not just deliberate schedule visits — a real increase in audit-log volume and character. This is accepted as the correct consequence of genuine reuse of an already-audited clinical-data function, not something to route around. (Contrast with `getPendingDeliveries`, Phase 18: deliberately never audited, because that data is operational, not clinical — a different precedent that doesn't apply here.)

## C. Pending lab orders

Calls `getPendingLabOrders(user)` — the exact function `/lab-orders/pending` already uses. Gated on `clinical.lab.results.enter` **as a capability check** (`hasCapability(user.role, 'clinical.lab.results.enter')`), not a hand-typed `[doctor, lab_staff]` list — the brief's own role list undercounts (`LAB_RESULTS_ENTER_ROLES` also includes `super_admin`), so gating on the capability itself is both correct and can't drift from what `getPendingLabOrders` already enforces internally. Same audit-on-every-view consequence as B (`lab_worklist_view`), same reasoning — accepted, not worked around.

Widget slices to the first 5 results — the function already returns ascending by `orderedAt` (oldest/most-overdue first), so "first 5" is naturally "most overdue 5," no new sort needed.

## D. Pending leave approvals

`leave/review/page.tsx`'s pending-request query only ever existed inline in that page component — there is no shared function to import, unlike B and C. Per explicit approval: new `src/lib/dashboard/pendingLeaveApprovals.ts`, replicating that page's exact query shape rather than modifying it:

```ts
export interface PendingLeaveApprovalRow {
  id: string
  staffName: string
  type: string
  startDate: string
  endDate: string
}

export async function getPendingLeaveApprovals(viewer: SessionUser): Promise<PendingLeaveApprovalRow[]>
```

- Gate: `hasCapability(viewer.role, 'hr.leave.approve')`, throwing `AuthError('Forbidden', 403)` — matching `leave/review/page.tsx`'s own `requireCapability('hr.leave.approve')`.
- Scoping: `viewer.role === 'branch_manager' ? where('branchId','==',viewer.branchId) : (no filter)` — copied verbatim from the review page's own scoping convention (deliberately `role === 'branch_manager'`, not `isBranchLocked`, matching what's actually there today).
- Same `status === 'pending'` filter, same `orderBy('createdAt', 'desc')`, same staff-name join pattern.
- Same Timestamp-safety discipline the review page already documents in its own comment ("Rows are built field-by-field from the raw doc (never spread) so a Firestore Timestamp can never leak") — `startDate`/`endDate` converted to `'YYYY-MM-DD'` strings, matching the review page's own `.toDate().toISOString().slice(0, 10)`.
- **No audit logging** — the review page itself never audited viewing pending requests, so the new function doesn't add one either. This isn't an omission; it's matching what's actually there.

`leave/review/page.tsx` itself is not modified.

## E. Widget components

Three new presentational components, matching Phase 23's established shape (no fetch calls, no state, receive already-fetched data as props):

- `UpcomingAppointmentsWidget` — `{ appointments: AppointmentRow[] }`
- `PendingLabOrdersWidget` — `{ orders: PendingLabOrderRow[] }`
- `PendingLeaveApprovalsWidget` — `{ requests: PendingLeaveApprovalRow[] }`

Each does its own `.slice(0, 5)` and shows the full array's `.length` as the count — none of the three reused/new functions return a pre-shaped `{ rows, totalCount }` the way Phase 23's four lib functions do (and reshaping `getAppointments`/`getPendingLabOrders` to do so would mean touching code this phase is explicitly not supposed to change), so the truncation/count responsibility lives in the presentational layer instead.

## F. Testing

`getAppointments`/`getPendingLabOrders` are already exercised elsewhere in this codebase (existing routes/pages) — no new tests for them, consistent with "reuse, don't reimplement" meaning the query logic isn't retested here either.

New coverage:
- A Vitest integration suite for `getPendingLeaveApprovals` (the one genuinely new function) — branch-scoped vs org-wide, `status: 'pending'` exclusion of approved/rejected requests, capability rejection.
- An explicit, named unit-test assertion that `nurse` does not hold `clinical.appointments.manage` — this is the one place this phase could easily get wrong by treating "dashboard visibility" as a lower bar than the capability it's actually gated on, so it gets checked directly rather than only inferred from the widget's conditional not rendering.
- Live verification (deferred to the plan/implementation stage, per this project's standing practice): confirm all three widgets render for a role holding the relevant capability and are absent for one that doesn't; confirm `nurse` specifically does not see the appointments widget; confirm Phase 23's four existing widgets are visually/behaviorally unchanged.

## Exit criteria

- All three widgets render correctly, each reusing an existing query (or, for leave approvals, an exact replica of one that only ever existed inline) rather than a new implementation.
- `nurse` explicitly confirmed not to see the appointments widget — both via a direct capability-matrix test and live verification, not inferred.
- Each widget's visibility matches its existing capability exactly, verified independently per widget.
- Phase 23's four existing widgets confirmed unchanged (code-level diff and live check).
- No new query logic was written where an existing one should have been reused — the completion report states explicitly which existing query (or, for leave approvals, which existing query it was copied from) backs each widget.
