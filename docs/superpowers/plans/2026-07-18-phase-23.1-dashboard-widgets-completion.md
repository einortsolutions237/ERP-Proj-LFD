## What shipped

Three more widgets on the same dashboard Phase 23 built: upcoming appointments, pending lab orders, pending leave approvals — each independently capability-gated, none touching a query it didn't already have a right to reuse.

- **Upcoming appointments** — calls `getAppointments({ upcomingOnly: true }, user)` (Phase 14, unmodified) directly. Gated on `clinical.appointments.manage`. `nurse` confirmed structurally excluded — both the dashboard's gate and `getAppointments`'s own internal re-check use the identical capability string, so there's no alternate path even hypothetically, and a direct unit-test assertion (`tests/unit/permissions.test.ts`) pins it going forward.
- **Pending lab orders** — calls `getPendingLabOrders(user)` (Phase 19.2, unmodified) directly. Gated on `clinical.lab.results.enter` as a capability check, not a hand-typed role list (the brief's own role list undercounted `super_admin`).
- **Pending leave approvals** — the one new function this phase, `getPendingLeaveApprovals` (`src/lib/dashboard/pendingLeaveApprovals.ts`), a byte-faithful replica of `leave/review/page.tsx`'s inline query (same `role === 'branch_manager'` scoping, same fields, same Timestamp-safety discipline). `leave/review/page.tsx` itself is untouched — this phase needed a *new* function because that query never had one to import, not because the existing one needed changing.

**Deliberate, accepted consequence, not a defect:** `getAppointments`/`getPendingLabOrders` both audit-log every call (`appointment_view`/`lab_worklist_view`). Reusing them means every dashboard load by a capable role now writes an audit entry, not just deliberate schedule/worklist visits — the correct behavior per this project's "every clinical read is audited" rule (load-bearing since Phase 13), independently confirmed at the final review to never leak onto the Phase 23 recent-activity feed (neither action is in that widget's whitelist).
