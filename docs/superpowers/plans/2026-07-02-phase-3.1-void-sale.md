# Phase 3.1 — Void a Sale Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let an authorized user (not `cashier`) undo a completed sale — full-sale void only — correctly reversing whatever stock it moved, with no payment/refund handling of any kind.

**Architecture:** Same modular monolith as Phases 1–3. No new collection — `sales` gains three fields (`voidedAt`/`voidedBy`/`voidReason`), `stockMovements` gains one new `type` value (`void`). One new endpoint, one new capability flip, no new Firestore rules (voiding stays Admin-SDK-only, same as every other write).

**Tech Stack:** Unchanged — Next.js App Router, TypeScript, Tailwind, Firebase Auth + Firestore + Admin SDK.

## Global Constraints

- Firestore rules default-deny; the existing `sales` rule (`allow write: if false`) already covers this — no rules changes needed, confirmed below
- Every write goes through an Admin-SDK-backed API route with an explicit field whitelist — never `{...body}` spread
- Firebase custom claims are the sole source of truth for authorization
- Enforce role + branch checks at UI and server for the flipped capability (rules don't need a matching change per the constraint above)
- Voiding produces a `sale_void` audit log entry via `writeAuditLog()`, with full details (original total, original line items, reason, the stock movements created to reverse it) — not a bare id
- No TDD / no automated test suite — manual verification against exit criteria, matching Phases 1–3
- This is full-sale void only: no partial/line-item returns, no refund/payment reversal, no PATCH beyond the void action itself — the original sale's `lineItems`/`payments`/`subtotal`/`total` are never modified

## Sale doc extension

Add three fields to `Sale` (`src/lib/types/sale.ts`):
```ts
voidedAt: FirebaseFirestore.Timestamp | null
voidedBy: string | null   // acting user's uid
voidReason: string | null
```
All `null` on a normal, active sale. Set together, exactly once, by the void endpoint — nothing else in the codebase ever writes to these three fields.

**Required amendment to the already-shipped `POST /api/sales`, flagging explicitly rather than silently touching reviewed code:** the create route's `tx.set(saleRef, {...})` call needs `voidedAt: null, voidedBy: null, voidReason: null` added to the object literal it writes. This is necessary for the same reason `transferId: null` is explicitly set on every non-transfer `stockMovements` entry — this codebase's established convention is to fully populate a document's shape at creation, not leave fields absent and rely on `undefined` reads elsewhere. Note (carried over from Phase 3's own finding): this codebase's Firestore collections are untyped `DocumentData`, so omitting these three fields would NOT be caught by `tsc` — it has to be gotten right by inspection, not by the compiler.

## Stock reversal mechanic — ledger-driven, not re-derived from `lineItems`

**Design choice, flagging explicitly (the Scope text's phrasing — "for every line item where type is product, reverse the stock" — could be read as "iterate `sale.lineItems`," but that's not what this plan does, for a specific reason):** the sale-creation transaction (Phase 3) deliberately aggregates multiple cart lines referencing the same product into **one** `stockMovements` entry per distinct product (the `normalizeCartLines()` mechanic that was a gated requirement of Phase 3). Voiding re-derives-from-`lineItems` would mean re-implementing that same aggregation a second time, risking the two aggregations drifting out of sync. Instead, voiding queries the **original `stockMovements` entries** this sale actually created (`where('saleId','==',id).where('type','==','sale')`) and reverses exactly those — one `void` entry per original `sale` entry, with the exact opposite (positive) `quantityDelta`. The ledger is already the authoritative record of what moved; re-aggregating from `lineItems` a second time is redundant and a source of potential drift, not a more faithful reversal.

One consequence: a service-only sale has zero `type:'sale'` `stockMovements` entries (Phase 3 design — services never touch stock), so the query returns empty and the void transaction correctly performs zero stock writes for it — satisfying "voiding a service-only sale produces zero stock movement activity" without any special-casing.

**No sufficiency check is needed on the write side of a void** — restoring stock is a pure additive `FieldValue.increment(+n)`; there is no lower bound to violate the way there is on a sale/adjustment/waste. This means the transaction doesn't need to read `productStock` at all before writing to it (unlike sale-creation, which must read-before-check-before-write for the negative-quantity guard).

**File:** Create `src/app/api/sales/[id]/void/route.ts`

Transaction shape, in order:
```
1. READ: the sale doc (saleRef.get())
2. READ: stockMovements where saleId == id AND type == 'sale' (tx.get on the query — both reads happen before any write)
3. Validate (throwing AuthError, caught after the reads, before any write):
   - sale doc doesn't exist -> 404
   - sale.voidedAt is already set -> 409 "Sale has already been voided"
   - acting user is branch_manager AND sale.branchId !== user.branchId -> 403 "Can only void sales for your own branch"
     (this check happens HERE, after reading the sale, not as an early pre-transaction guard like the stock routes'
     branch check — the sale's branchId isn't known until it's read, since the client only supplies the sale id via
     the URL, not a branchId. This is a deliberate, necessary difference from the stock routes' structure, not a
     lesser version of the same check.)
4. WRITE: tx.update(saleRef, { voidedAt: new Date(), voidedBy: user.uid, voidReason: reason })
5. For each original 'sale' movement found in step 2:
   - tx.set(productStock doc for that movement's productId+branchId, { quantity: FieldValue.increment(-movement.quantityDelta), updatedAt: new Date() }, { merge: true })
     (movement.quantityDelta is negative from the original sale; negating it yields the correct positive restoration amount)
   - tx.set(new stockMovements doc, { productId: movement.productId, branchId: movement.branchId, type: 'void',
     quantityDelta: -movement.quantityDelta, reason: null, actorUid: user.uid, createdAt: new Date(),
     transferId: null, saleId: id })
```

**Request body:** `{ reason: string }` — required, non-empty after trim. 400 otherwise.

**Capability:** `requireCapability('pos.sale.void')` gates the whole route (see Permissions below — this alone keeps `cashier` out).

**Audit log**, after the transaction commits:
```ts
await writeAuditLog({
  action: 'sale_void',
  actorUid: user.uid,
  actorEmail: user.email,
  targetUid: id,
  branchId: sale.branchId,
  details: {
    originalTotal: sale.total,
    originalLineItems: sale.lineItems,
    reason,
    reversalMovements: <the productId/branchId/quantityDelta of every void movement just created>,
  },
})
```

**Response:** `{ ok: true }`, 200 (this is a state-transition on an existing resource, not a creation — 200, not 201).

## `stockMovements` extension

Add `'void'` to `StockMovementType` in `src/lib/types/stock.ts`:
```ts
export type StockMovementType = 'restock' | 'adjustment' | 'waste' | 'transfer_out' | 'transfer_in' | 'sale' | 'void'
```
No new field on `StockMovement` — `saleId` already exists (Phase 3) and is reused for `void` movements to link back to the sale being voided (same field, same meaning: "which sale is this movement associated with").

**No change.** The existing `sales` rule (`allow write: if false`) already blocks every client write path, including a void — voiding goes through the Admin SDK exactly like sale creation. Confirmed per the user's own scope text ("rules don't need to change").

## UI

**File:** Create `src/components/pos/VoidSaleButton.tsx` (client component)
- Props: `saleId: string`.
- Renders a "Void sale" button. Clicking it reveals (inline, no modal library needed — matches this codebase's existing pattern of inline expansion for the Phase 2 stock adjust/transfer forms) a required `reason` textarea and a "Confirm void" button, plus a "Cancel" to collapse it back without submitting.
- On confirm: `POST /api/sales/{saleId}/void` with `{reason: reason.trim()}`. Disabled until `reason.trim().length > 0`. On success, `router.refresh()` (server re-fetches the now-voided sale). On failure, surface `body.error`, leave the reason text intact — same no-silent-retry posture Phase 3's checkout form used.

**File: `src/app/(dashboard)/pos/sales/[id]/page.tsx`** (modify, existing server component)
- Fetch already reads the full sale doc — no new fetch needed, `voidedAt`/`voidedBy`/`voidReason` come along for free once the type is extended.
- If `data.voidedAt` is set: render a "Voided" badge near the top (e.g. `<span className="...bg-red-100 text-red-700...">Voided</span>`) plus a small line showing when/by-whom/why (`voidedBy` — a raw uid is acceptable, matching `cashierUid`'s existing display convention; `voidReason` shown verbatim).

- If `data.voidedAt` is NOT set AND the user lacks the capability: render nothing extra — no void control, no placeholder, nothing a `cashier` (or any non-void-capable role) can see that hints the feature exists.

**File: `src/app/(dashboard)/pos/sales/page.tsx`** (modify, existing server component) + **`src/components/pos/SalesTable.tsx`** (modify)
- The list page computes `voided: boolean` per row as `doc.data().voidedAt != null` (a plain boolean, not the raw Timestamp — simpler than adding another Timestamp→ISO-string conversion for a value the list view only needs as a yes/no signal). Add `voided: boolean` to `SaleRow`.
- `SalesTable` renders a "Voided" badge (or plain "Voided" text in the existing table, styled distinctly, e.g. red) in a new/adjusted column when `row.voided` is true — active sales show nothing extra there. No exclusion/filtering of voided sales from the list — they stay visible, just marked (per the exit criteria and the explicit "don't build reporting exclusions" scope boundary).

## Open questions / judgment calls (flagging per project convention, not guessing)

- **Ledger-driven reversal instead of re-deriving from `lineItems`** — flagged in detail above; this is the one place this plan's mechanism differs from a literal reading of the Scope text, and the reasoning (avoiding duplicated aggregation logic, ledger-as-source-of-truth) is the same principle Phase 3's own design already established.
- **Branch check happens after the transactional read, not as an early guard** — flagged above; a necessary structural difference from the stock routes, not a weaker check.
- **`POST /api/sales/[id]/void` returns 200, not 201** — this is a state transition on an existing resource, not a creation; flagging the convention choice since the plan doesn't have a prior "void-like" endpoint to copy exactly.
- Partial/line-item returns, actual payment/refund reversal, and reporting/dashboards excluding voided sales are explicitly out of scope per your instruction — not touched anywhere in this plan.
