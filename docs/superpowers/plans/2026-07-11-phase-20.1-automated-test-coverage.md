# Phase 20.1 — Automated Test Coverage Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up this project's first automated test suite — role×capability matrix, `canMessage` matrix, the sale transaction, `isBranchLocked` read scoping, and the clinical wall — running against the Firestore/Auth emulators, so these five bug classes (all drawn from this project's actual history) get a regression net instead of relying solely on manual phase-by-phase audit.

**Architecture:** Vitest as the runner. Two tiers of tests: pure unit tests (role×capability, `canMessage`) that import the real constants/functions with zero I/O, and integration tests (sale transaction, branch scoping, clinical wall) that invoke Next.js route handlers' exported `GET`/`POST` functions directly as plain async functions against the Firestore + Auth emulators, with `next/headers`'s `cookies()` mocked per-test to inject a real emulator-minted session cookie. `firebase emulators:exec` wraps `npm test` so a clean checkout needs nothing pre-running.

**Tech Stack:** Vitest 2.x, `firebase-tools` (already available via `npx`, confirmed v15.22.4 in this environment) for the emulator, `firebase-admin` (already a dependency) for fixture seeding and session-cookie minting.

## Global Constraints

- Tests run only against the Firestore/Auth emulators — never live `erp-lfd`. No test may read or write production data.
- Zero application-code changes beyond `package.json` and new test-config/test files. If a test surfaces a real bug, flag it in the completion report per the known-issues policy; fix it inline only if small and obvious, called out explicitly as a deviation.
- Fixtures are seeded fresh per test file and the emulator's Firestore/Auth state is cleared between test files (not between individual tests within a file, to keep the suite fast) — each test file is responsible for creating exactly the data it needs and must not assume another file's leftovers.
- No live-data writes, no `erp-lfd` credentials read, no network calls outside `localhost`.

## Design decisions (the "send the plan, wait for go-ahead" content)

**1. Runner: Vitest.** Native ESM/TS support with no transform config, fast startup/watch, and it's the tool this Next.js 16 App Router stack would reach for by default (Jest's ESM story requires extra `transform`/`moduleNameMapper` wiring this repo doesn't have and doesn't need otherwise). `vitest.config.ts` at the repo root, `environment: 'node'` (no DOM needed — every test here is server-side), path alias `@` → `src` mirrored from `tsconfig.json`.

**2. Route invocation: direct handler call, not HTTP.** Every route handler under test (`GET`, `POST` exports in `src/app/api/**/route.ts`) is a plain async function taking a Web-standard `Request` (and, for dynamic routes, a `{ params: Promise<...> }` second argument) and returning a Web-standard `Response`. Node 20+ (which Next 16 requires) has `Request`/`Response` as globals, so a test can call `await POST(new Request('http://localhost/api/sales', { method: 'POST', body: JSON.stringify(...) }))` directly — no dev server, no port, no HTTP client.

The one obstacle: `requireCapability()` → `getSessionUser()` calls `cookies()` from `next/headers`, which needs Next's request-scoped `AsyncLocalStorage` to be active. That context doesn't exist outside a real Next request lifecycle. Fix: mock the whole `next/headers` module per test file with `vi.mock('next/headers', ...)`, backed by a tiny module-level variable holding "the current test's session cookie value." This is the standard, documented pattern for unit-testing App Router route handlers and needs zero changes to `server-guard.ts` or any route file.

**3. Fixture seeding / auth.** Mirrors `scripts/seed.ts`'s existing pattern exactly (already checked into this repo, already proven against real custom claims):
- `auth.createUser({ email, password })` + `auth.setCustomUserClaims(uid, { role, branchId })` for each test staff account.
- A session cookie is minted the same way Phase 18's live verification did it (per CLAUDE.md: "sessions ... minted via Admin-SDK custom-token exchange"): `auth.createCustomToken(uid)` → exchange for an ID token via the Auth emulator's `identitytoolkit` REST endpoint (`signInWithCustomToken`, pointed at `http://{FIREBASE_AUTH_EMULATOR_HOST}/identitytoolkit.googleapis.com/v1/accounts:signInWithCustomToken?key=fake-key`) → `auth.createSessionCookie(idToken, { expiresIn })`. Wrapped once in `tests/setup/fixtures.ts` as `mintSessionCookie(uid)`, reused by every integration suite.
- Firestore fixtures (branches, products, `productStock`, customers) are plain Admin SDK `.set()` calls against the emulator — no rules to satisfy since Admin SDK always bypasses Firestore security rules.

**4. Suite list** (file → what it covers, no I/O vs. emulator):
| File | Tier | Covers |
|---|---|---|
| `tests/unit/permissions.test.ts` | pure | Priority 1 — full role×capability snapshot + structural invariants |
| `tests/unit/canMessage.test.ts` | pure | Priority 2 — full `canMessage` truth table |
| `tests/integration/sales.test.ts` | emulator | Priority 3 — sale transaction, all 8 named behaviors |
| `tests/integration/branch-scoping.test.ts` | emulator | Priority 4 — `isBranchLocked` read scoping on sales/staff/departments/stock |
| `tests/integration/clinical-wall.test.ts` | emulator | Priority 5 — clinical wall positive/negative pairs |

**5. `firebase.json` gains an `emulators` block** (ports 8080 Firestore / 9099 Auth / 4000 UI disabled for CI-friendliness) and a `demo-erp-lfd` project is used for the emulator (Firebase's `demo-` prefix convention: the CLI runs the emulator in pure offline mode, no real project or billing account needed, and refuses any accidental production traffic by construction — this is *why* it's the right choice here, not just a naming convention).

**6. `.env.test`** holds `FIREBASE_ADMIN_PROJECT_ID=demo-erp-lfd`, a syntactically-valid-but-inert dummy `FIREBASE_ADMIN_CLIENT_EMAIL`/`FIREBASE_ADMIN_PRIVATE_KEY` (needed only so `cert()`'s PEM-format check in `src/lib/firebase/admin.ts` doesn't throw at construction — the key is never used to sign anything real, since `FIRESTORE_EMULATOR_HOST`/`FIREBASE_AUTH_EMULATOR_HOST` make the Admin SDK talk to the local emulator without ever invoking credential-based signing), plus `FIRESTORE_EMULATOR_HOST=127.0.0.1:8080` and `FIREBASE_AUTH_EMULATOR_HOST=127.0.0.1:9099`. Loaded by `vitest.config.ts` via `envDir`/`setupFiles`, not by `dotenv` in application code — no change to how the app itself reads env vars.

**7. `package.json` scripts:**
```json
"test": "firebase emulators:exec --only firestore,auth --project demo-erp-lfd \"vitest run\"",
"test:watch": "vitest"
```
`npm test` is fully self-contained (starts both emulators, runs the suite once, tears down, correct exit code) — this is what "clean checkout, no dependency on prior state" means operationally. `npm run test:watch` assumes emulators are already running in another terminal (`npx firebase emulators:start --only firestore,auth --project demo-erp-lfd`); this is documented as a one-line comment in `package.json`'s neighboring `README`/plan, not enforced in code, since `emulators:exec` cannot host a long-lived watch process.

```typescript
let activeCookie: string | null = null

export function mockNextHeaders() {
  vi.mock('next/headers', () => ({

    }),
  }))

export async function withSession<T>(sessionCookie: string | null, fn: () => Promise<T>): Promise<T> {
  const previous = activeCookie
  activeCookie = sessionCookie
  try {
    return await fn()
  } finally {
    activeCookie = previous
  }
}
```

```typescript
import { describe, it, expect } from 'vitest'
import {
  ROLES,
  ROLE_CAPABILITIES,
  BRANCH_LOCKED_ROLES,
  hasCapability,
  isBranchLocked,
  type Capability,
} from '@/lib/auth/permissions'

const CAPABILITIES = Object.keys(ROLE_CAPABILITIES) as Capability[]

describe('role x capability matrix', () => {
  it('matches the exact grant table for every role x every capability', () => {
    const matrix: Record<string, Record<string, boolean>> = {}
    for (const role of ROLES) {
      matrix[role] = {}
      for (const capability of CAPABILITIES) {
        matrix[role][capability] = hasCapability(role, capability)
      }
    }
    expect(matrix).toMatchSnapshot()
  })

  it('super_admin holds every capability', () => {
    for (const capability of CAPABILITIES) {
      expect(hasCapability('super_admin', capability)).toBe(true)
    }
  })

  it('clinical.record.create is exactly [super_admin, doctor]', () => {
    expect(ROLE_CAPABILITIES['clinical.record.create'].slice().sort()).toEqual(['doctor', 'super_admin'])
  })

  it('general_manager does not hold crm.customer.view (no commercial view)', () => {
    expect(hasCapability('general_manager', 'crm.customer.view')).toBe(false)
  })

  it('general_manager does not hold clinical.record.create (no clinical authoring)', () => {
    expect(hasCapability('general_manager', 'clinical.record.create')).toBe(false)
  })

  it('general_manager does not hold seminars.attendance.record (no attendance recording)', () => {
    expect(hasCapability('general_manager', 'seminars.attendance.record')).toBe(false)
  })

  it('branch-locked roles are exactly [branch_manager, cashier, inventory_manager]', () => {
    expect(BRANCH_LOCKED_ROLES.slice().sort()).toEqual(['branch_manager', 'cashier', 'inventory_manager'])
    for (const role of ROLES) {
      expect(isBranchLocked(role)).toBe(BRANCH_LOCKED_ROLES.includes(role))
    }
  })

  // CORRECTED post-implementation (2026-07-12): the plan's original version
  // of this test asserted admin holds exactly the five Phase 17 system/
  // access-administration capabilities and no clinical capability at all.
  // Both were wrong — admin genuinely holds 9 capabilities: the five Phase
```

## Task 3: `canMessage` truth table (Priority 2)

**Files:**

**Interfaces:**

```typescript
import { describe, it, expect } from 'vitest'
import { canMessage, type MessagingParty } from '@/lib/messaging/canMessage'
import { ROLES, type RoleId } from '@/lib/auth/permissions'

const BRANCH_A = 'branch-a'
const BRANCH_B = 'branch-b'

function party(uid: string, role: RoleId, branchId: string): MessagingParty {
  return { uid, role, branchId }
}

// Expected reachability, independent of the implementation under test —
// hand-derived from CLAUDE.md's Phase 19 hierarchy description.
function expected(aRole: RoleId, bRole: RoleId, sameBranch: boolean): boolean {
  if (aRole === 'super_admin' || bRole === 'super_admin') return true
  if (aRole === 'it_admin' || bRole === 'it_admin') return true
  const hierarchyExempt: RoleId[] = ['branch_manager', 'general_manager', 'it_admin', 'super_admin']
  const aGeneric = !hierarchyExempt.includes(aRole)
  const bGeneric = !hierarchyExempt.includes(bRole)
  if (aGeneric && bRole === 'branch_manager' && sameBranch) return true
  if (bGeneric && aRole === 'branch_manager' && sameBranch) return true
  if (aRole === 'branch_manager' && bRole === 'general_manager') return true
  if (bRole === 'branch_manager' && aRole === 'general_manager') return true
  return false
}

describe('canMessage — exhaustive role-pair x branch-sameness truth table', () => {
  for (const aRole of ROLES) {
    for (const bRole of ROLES) {
      for (const sameBranch of [true, false]) {
        it(`${aRole} (${sameBranch ? 'same' : 'diff'} branch) -> ${bRole}`, () => {
          const a = party('uid-a', aRole, BRANCH_A)
          const b = party('uid-b', bRole, sameBranch ? BRANCH_A : BRANCH_B)
          expect(canMessage(a, b)).toBe(expected(aRole, bRole, sameBranch))
          expect(canMessage(b, a)).toBe(expected(aRole, bRole, sameBranch)) // symmetry
        })
      }
    }
  }

  it('self-message is always false, even for super_admin/it_admin', () => {
    const superAdmin = party('same-uid', 'super_admin', BRANCH_A)
    expect(canMessage(superAdmin, { ...superAdmin })).toBe(false)
    const itAdmin = party('same-uid-2', 'it_admin', BRANCH_A)
    expect(canMessage(itAdmin, { ...itAdmin })).toBe(false)
  })

  it('branch_manager <-> branch_manager is false, even in the same branch', () => {
    const a = party('uid-a', 'branch_manager', BRANCH_A)
    const b = party('uid-b', 'branch_manager', BRANCH_A)
    expect(canMessage(a, b)).toBe(false)
  })

  it('general_manager -> staff (cashier) is false, in any branch', () => {
    const gm = party('uid-gm', 'general_manager', BRANCH_A)
    const cashierSame = party('uid-c1', 'cashier', BRANCH_A)
    const cashierDiff = party('uid-c2', 'cashier', BRANCH_B)
    expect(canMessage(gm, cashierSame)).toBe(false)
    expect(canMessage(gm, cashierDiff)).toBe(false)
  })

  it('cross-branch staff -> branch_manager is false', () => {
    const staff = party('uid-s', 'cashier', BRANCH_A)
    const foreignManager = party('uid-m', 'branch_manager', BRANCH_B)
    expect(canMessage(staff, foreignManager)).toBe(false)
  })

  it('it_admin reaches and is reached by every other role, both directions', () => {
    const itAdmin = party('uid-it', 'it_admin', BRANCH_A)
    for (const role of ROLES) {
      if (role === 'it_admin') continue
      const other = party('uid-other', role, BRANCH_B)
      expect(canMessage(itAdmin, other)).toBe(true)
      expect(canMessage(other, itAdmin)).toBe(true)
    }
  })
})
```
