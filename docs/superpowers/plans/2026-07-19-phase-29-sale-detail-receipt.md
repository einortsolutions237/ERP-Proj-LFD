# Phase 29 — Sale Detail View & Printable Receipt Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let any past sale be opened from the Sales Log into a full detail view (line items, payments, customer/branch/cashier, void status, pending deliveries), and print/save it as a clean receipt via the browser's native print dialog.

**Architecture:** A new server-side data helper (`getSaleDetail`) resolves a raw `sales` doc into a fully display-ready shape (names instead of bare IDs, ISO strings instead of Timestamps), reusing the exact belt-and-suspenders capability-recheck pattern already established by `getPendingDeliveries`/`getAppointments`/`getLabRecords`. The existing (but pre-design-system, and subtly buggy) `pos/sales/[id]/page.tsx` is rewritten to consume it, gains a print-only receipt block using Tailwind's `print:` variant (`@media print` under the hood), and a tiny client-only "Print" button. `NavShell.tsx` gets minimal additive `print:hidden` classes so the sidebar/header don't appear in the printed output — no new library, no server-side PDF generation, no new capability, zero changes to `api/sales/route.ts`.

**Tech Stack:** Next.js App Router (Server Component page + one small `'use client'` button), Firebase Admin SDK (Firestore reads only), Tailwind CSS `print:` variant, Vitest against the Firestore/Auth emulators (this project's existing test harness — there is no component-rendering test framework, per Phase 27's note, so UI correctness here is verified by build + live/manual check, not a unit test).

## Global Constraints

- No new Firebase custom claim, no new capability — reuse `pos.sale.view` exactly as it gates the Sales Log today (`CASHIER_BRANCH_MGR` = `['super_admin', 'branch_manager', 'cashier']`).
- Branch-scoping must match `GET /api/sales`/the Sales Log page exactly: `isBranchLocked(user.role)` gates visibility, not an unconditional `branchId` comparison.
- Zero changes to `src/app/api/sales/route.ts`, the void route, or any stock/payment transaction logic anywhere.
- No new npm dependency. Printing is `window.print()` + `@media print` CSS only.
- No receipt branding/customization system — a static "LFD Services" header (the name already used verbatim on the login page) is fine; a configurable logo/template feature is not in scope.
- No bulk export, no emailing a receipt — single-sale, on-demand print only.

## Investigation findings (read this before touching anything)

1. **A sale detail page already exists** at `src/app/(dashboard)/pos/sales/[id]/page.tsx`, linked from `SalesTable.tsx`'s existing "View" column. It is **pre-design-system** (raw `border-collapse` tables, `text-gray-500`, `rounded bg-red-100`) and has a **real, live branch-scoping bug**: line 25 does `if (data.branchId !== user.branchId) notFound()` **unconditionally**, for every role — including `super_admin`, which the Sales Log (Phase 20) correctly lets see every branch's sales via `isBranchLocked(user.role)`. Today, a `super_admin` clicking "View" on a foreign-branch sale from their own Sales Log gets a false 404. This phase touches this exact file for new functionality anyway, so per this project's Known-issues policy, **the fix is in scope and folded into Task 1/2**, not just re-flagged.
2. There is no `GET /api/sales/[id]` route — only the void route (`api/sales/[id]/void/route.ts`) looks up a sale by ID, inside its transaction. The plain-read path is the Server Component page reading Firestore directly (matching how `SalesLogPage`, `ExpensesPage`, etc. all do their own direct reads) — no new API route is needed or added.
3. `pos.sale.view` roles (`CASHIER_BRANCH_MGR` = `super_admin`/`branch_manager`/`cashier`) are a strict subset of `pos.delivery.fulfill` roles (`POS_DELIVERY_FULFILL_ROLES` = `super_admin`/`general_manager`/`branch_manager`/`cashier`). So showing "pending deliveries generated by this sale" to anyone who already holds `pos.sale.view` **does not widen access** beyond what `pos.delivery.fulfill` already implies for those same roles — no extra capability check is needed on that section, and none is added.
4. Viewing a sale (list or detail) has never been audit-logged anywhere in this app (same precedent `getPendingDeliveries.ts`'s own comment documents for stock/sales-adjacent reads) — this phase does not add one.
5. `NavShell.tsx` (wraps every page: `Sidebar` + header + `<main>`) has no print handling today, and neither does any other file (`grep` for `print:`/`@media print` across `src/` returns nothing) — this is a new mechanic for the app, not an extension of an existing one.
6. Tailwind's `print:` variant is a core variant (maps to `@media print`), available with zero config changes on this project's Tailwind version.

```typescript
import { getAdminFirestore } from '@/lib/firebase/admin'
import { hasCapability, isBranchLocked } from '@/lib/auth/permissions'
import { AuthError, type SessionUser } from '@/lib/auth/server-guard'
import type { Sale, SaleLineItem, SalePayment } from '@/lib/types/sale'
import type { PendingDelivery, PendingDeliveryStatus } from '@/lib/types/pendingDelivery'

export interface SaleDetailPendingDelivery {
  id: string
  productId: string
  productName: string
  quantityOwed: number
  status: PendingDeliveryStatus
  fulfilledByName: string | null
  fulfilledAt: string | null
}

export interface SaleDetail {
  id: string
  branchId: string
  branchName: string
  createdAt: string
  cashierUid: string
  cashierName: string
  customerId: string | null
  customerName: string | null
  lineItems: SaleLineItem[]
  subtotal: number
  discountAmount: number
  taxAmount: number
  total: number
  payments: SalePayment[]
  voided: boolean
  voidedAt: string | null
  voidedBy: string | null
  voidedByName: string | null
  voidReason: string | null
  pendingDeliveries: SaleDetailPendingDelivery[]
}

// Called by /pos/sales/[id] — the same belt-and-suspenders capability
// recheck as getPendingDeliveries/getAppointments/getLabRecords, even though
// the page itself already gates on pos.sale.view before calling this.
//
// Returns null for both "doesn't exist" and "exists in a branch this
// branch-locked viewer can't see" — deliberately indistinguishable, same as
// the "don't reveal a sale exists in another branch" behavior this replaces,
// now keyed off isBranchLocked() instead of an unconditional branchId
// comparison, matching GET /api/sales's Phase 20 fix.
export async function getSaleDetail(saleId: string, viewer: SessionUser): Promise<SaleDetail | null> {
  if (!hasCapability(viewer.role, 'pos.sale.view')) {
    throw new AuthError('Forbidden', 403)
  }

  const db = getAdminFirestore()
  const saleSnap = await db.collection('sales').doc(saleId).get()
  if (!saleSnap.exists) return null

  const sale = saleSnap.data() as Sale
  if (isBranchLocked(viewer.role) && sale.branchId !== viewer.branchId) return null

  const deliveriesSnap = await db.collection('pendingDeliveries').where('saleId', '==', saleId).get()
  const deliveryDocs = deliveriesSnap.docs.map((d) => ({ id: d.id, data: d.data() as PendingDelivery }))

  const uniqueProductIds = Array.from(new Set(deliveryDocs.map((d) => d.data.productId)))
  const uniqueStaffUids = Array.from(
    new Set(
      [sale.cashierUid, sale.voidedAt ? sale.voidedBy : null, ...deliveryDocs.map((d) => d.data.fulfilledBy)].filter(
        (uid): uid is string => uid !== null && uid !== undefined
      )
    )
  )

  const [branchDoc, customerDoc, productDocs, staffDocs] = await Promise.all([
    db.collection('branches').doc(sale.branchId).get(),
    sale.customerId ? db.collection('customers').doc(sale.customerId).get() : Promise.resolve(null),
    Promise.all(uniqueProductIds.map((id) => db.collection('products').doc(id).get())),
    Promise.all(uniqueStaffUids.map((uid) => db.collection('staff').doc(uid).get())),
  ])

  const productNames: Record<string, string> = {}
  uniqueProductIds.forEach((id, i) => {
    productNames[id] = (productDocs[i].data()?.name as string | undefined) ?? id
  })
  const staffNames: Record<string, string> = {}
  uniqueStaffUids.forEach((uid, i) => {
    staffNames[uid] = (staffDocs[i].data()?.name as string | undefined) ?? uid
  })

  return {
    id: saleId,
    branchId: sale.branchId,
    branchName: (branchDoc.data()?.name as string | undefined) ?? sale.branchId,
    createdAt: sale.createdAt.toDate().toISOString(),
    cashierUid: sale.cashierUid,
    cashierName: staffNames[sale.cashierUid] ?? sale.cashierUid,
    customerId: sale.customerId,
    customerName: sale.customerId ? ((customerDoc?.data()?.name as string | undefined) ?? sale.customerId) : null,
    lineItems: sale.lineItems,
    subtotal: sale.subtotal,
    discountAmount: sale.discountAmount,
    taxAmount: sale.taxAmount,
    total: sale.total,
    payments: sale.payments,
    voided: sale.voidedAt != null,
    voidedAt: sale.voidedAt ? sale.voidedAt.toDate().toISOString() : null,
    voidedBy: sale.voidedBy,
    voidedByName: sale.voidedBy ? (staffNames[sale.voidedBy] ?? sale.voidedBy) : null,
    voidReason: sale.voidReason,
    pendingDeliveries: deliveryDocs.map(({ id, data }) => ({
      id,
      productId: data.productId,
      productName: productNames[data.productId] ?? data.productId,
      quantityOwed: data.quantityOwed,
      status: data.status,
      fulfilledByName: data.fulfilledBy ? (staffNames[data.fulfilledBy] ?? data.fulfilledBy) : null,
      fulfilledAt: data.fulfilledAt ? data.fulfilledAt.toDate().toISOString() : null,
    })),
  }
}
```

          <div className="flex justify-between">
            <span className="text-slate">Subtotal</span>
            <span className="font-mono text-ink">{sale.subtotal.toFixed(2)}</span>

                    </th>
                  </tr>

                  {sale.pendingDeliveries.map((d) => (

                            {d.fulfilledByName ? `by ${d.fulfilledByName}` : ''}
                            {d.fulfilledAt ? ` on ${new Date(d.fulfilledAt).toLocaleString()}` : ''}
                          </span>
                        )}
                      </td>
                    </tr>
                  ))}
                </tbody>
              </table>
            </div>
          </div>
        )}
      </div>

      {/* ---- Print-only receipt (invisible on screen, the only thing that
          prints). Plain black-on-white — receipts print on B/W printers, so
          this deliberately does not use the ink/marine/etc. color tokens. ---- */}
      <div className="hidden print:block print:p-6 print:text-black">
        <div className="mx-auto max-w-sm space-y-3 text-sm">
          <div className="text-center">
            <p className="text-base font-semibold">LFD Services</p>
            <p>{sale.branchName}</p>
            <p>{createdAtDisplay}</p>
          </div>
          {sale.voided && <p className="text-center font-semibold">*** VOIDED ***</p>}
          <div className="border-t border-black/40 pt-2">
            {sale.lineItems.map((line, i) => (
              <div key={i} className="flex justify-between">
                <span>
                  {line.name} × {line.quantity}
                </span>
                <span className="font-mono">{line.lineTotal.toFixed(2)}</span>
              </div>
            ))}
          </div>
          <div className="space-y-1 border-t border-black/40 pt-2">
            <div className="flex justify-between">
              <span>Subtotal</span>
              <span className="font-mono">{sale.subtotal.toFixed(2)}</span>
            </div>
            <div className="flex justify-between">
              <span>Discount</span>

            <div className="flex justify-between">
              <span>Tax</span>

            </div>
            <div className="flex justify-between font-semibold">
              <span>Total</span>
              <span className="font-mono">{sale.total.toFixed(2)}</span>
            </div>
          </div>
          <div className="space-y-1 border-t border-black/40 pt-2">
            {sale.payments.map((p, i) => (
              <div key={i} className="flex justify-between">
                <span>
                  {p.method}
                  {p.reference ? ` (${p.reference})` : ''}
                </span>
                <span className="font-mono">{p.amount.toFixed(2)}</span>
              </div>
            ))}

          <div className="border-t border-black/40 pt-2 text-xs">

            {sale.customerName && <p>Customer: {sale.customerName}</p>}
          </div>
        </div>
      </div>
    </div>
  )
}
